# パフォーマンス最適化バックログ

## 背景

ユーザーからプロンプト送信から回答表示までの処理が多すぎるとの指摘あり。
分析の結果、以下の最適化案を特定したが、即座の実装は見送り。

---

## 最適化案2: システムプロンプトキャッシュ

### 概要
`buildSystemPrompt()` のDBアクセス結果をキャッシュし、50〜200msのレイテンシ削減を目指す。

### メリット
- DB負荷軽減
- レイテンシ削減（50〜200ms）
- スケーラビリティ向上

### デメリット・リスク
- 古いプロンプトが使われる可能性（TTL管理が必要）
- Vercel複数インスタンスでのキャッシュ不整合
- 開発時の混乱（反映遅延）

### 実装案
```typescript
const promptCache = new Map<string, { prompt: string; expires: number }>();
```

### 検討事項
- [ ] キャッシュTTLの適切な値（5分？）
- [ ] 開発環境では無効化する仕組み
- [ ] 必要に応じてRedis移行の検討

---

## 最適化案3: システムプロンプト軽量化

### 概要
`programId !== "all"` の場合、該当番組のみの情報をプロンプトに含める。
全番組（~300行）→ 1番組（~20行）に削減。

### メリット
- トークン数大幅削減
- LLMコスト削減（入力課金の場合）
- レスポンス品質向上（ノイズ削減）

### デメリット・リスク
- 番組横断的な質問に弱くなる（「マツコと千鳥の共通点は？」）
- 破壊的変更の可能性
- "all"選択時は依然として全情報が必要

### 実装案
```typescript
const programs = programId === "all" 
  ? PROGRAMS 
  : PROGRAMS.filter(p => p.id === programId);
```

### 検討事項
- [ ] 複数番組指定の仕様検討
- [ ] 既存機能への影響調査
- [ ] 段階的ロールアウト計画

## 最適化案4: 要約処理の並列化/遅延化

### 概要
`ClientMemory` の要約処理を非同期化し、ストリーム開始をブロックしない。

### メリット
- 最大のレイテンシ削減（1〜3秒）
- 即座の応答開始
- リソース効率化（並列実行）

### デメリット・リスク
- ⚠️ **要約が間に合わない場合のフォールバック必要**
- ⚠️ **コスト増加（要約+ストリーミング同時実行）**
- ⚠️ **実装が複雑（状態管理・エラーハンドリング）**
- ⚠️ **トークン浪費の可能性**

### 実装案
```typescript
if (shouldSummarize) {
  updateSummary().then(() => { /* 次のターンで使用 */ });
}
await streamLLMResponse(...);  // 要約を待たない
```

### 検討事項
- [ ] 要約完了前のフォールバック動作設計
- [ ] コスト影響の試算
- [ ] エッジケースの網羅的テスト

## 実装判断基準

これらの最適化を検討するタイミング：

1. **メトリクス収集後** - 現状のボトルネックを計測してから
2. **ユーザーからの苦情** - 実際に速度問題が報告された場合
3. **スケール時** - ユーザー数・トラフィック増加時
