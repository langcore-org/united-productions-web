# チャットUX遅延問題調査レポート

## 問題の現状

ユーザーのスクリーンショットから確認された問題：
- **送信後、数秒間何も表示されない**
- その後、 suddenly に「Web検索」などのツール使用表示が複数出現
- この「無反応期間」がUX上の大きな問題

```
[ユーザー送信]
     ↓
     ~~~~ 数秒間、画面が何も動かない ~~~~ ← 問題の核心
     ↓
[Web検索] マツコの知らない世界 高視聴率...
[Web検索] "マツコの知らない世界" 視聴率 最高
[Web検索] マツコの知らない世界 視聴率ランキング...
```

| ステップ | 処理内容 | 推定時間 | 改善可能性 |
|---------|---------|---------|-----------|

| **⑤ ストリーミング** | コンテンツ受信・表示 | 継続的 | - |

## 根本原因

### 問題1: xAI APIの「思考時間」が長い（主要因）

xAI APIは、**ツールを使用する場合**、以下の処理をサーバーサイドで行う：

1. **ツール使用の判断**（数百ms）
2. **実際のツール実行**（web_search等、1-3秒）
3. **結果を踏まえた回答生成**（数百ms）

この間、SSE接続は確立しているが、**イベントが送信されない**。つまり：

```
[アプリ] → fetch開始 ────────────────→ [xAIサーバー]
            ↓                           ↓
         即座に接続確立              ツール実行中
            ↓                           ↓
         イベント待ち               イベント未送信
            ↓                           ↓
         「何も表示されない」         処理継続中
```

### 問題2: UIが「何もない状態」を表示しない（助長因）

`StreamingSteps.tsx` は `hasAnyContent` が true でないと**何もレンダリングしない**：

```typescript
// StreamingSteps.tsx（現在）
const hasAnyContent =
  completedSummarizations.length > 0 ||
  completedToolCalls.length > 0 ||
  runningSummarizations.length > 0 ||
  runningToolCalls.length > 0 ||
  !!content ||
  citations.length > 0;

return (
  <div>
    {hasAnyContent && (  // ← falseの間、何も表示されない
      <div>...ヘッダーとメッセージ...</div>
    )}
  </div>
);
```

**`phase === "preparing"` や `phase === "streaming"` の初期段階でも、**
**`hasAnyContent` は false のまま**。

## 改善策

### 案A: 「入力中...」スケルトンの即座表示（推奨・低コスト）

`phase === "streaming"` になった直後（xAI API接続確立後）、
**最初のイベントが届くまで**スケルトンを表示する。

```typescript
// StreamingSteps.tsx
{(phase === "streaming" && !hasAnyContent) && (
  <SkeletonMessage />  // 「入力中...」のスケルトン
)}
```

**効果**: xAI APIの「思考時間」中も「入力中...」が表示され、
ユーザーは「動いている」と感じる。

**実装工数**: 30分〜1時間

---

### 案B: 接続ステータスの詳細表示（推奨・中コスト）

`streaming` フェーズを細分化：

```typescript
type StreamingSubPhase = 
  | "connecting"      // xAI API接続確立
  | "thinking"        // LLMが回答を考え中
  | "tool_executing"  // ツール実行中
  | "responding";     // 回答生成中
```

UI表示：
- 「接続中...」→「考え中...」→「検索中...」→「回答生成中...」

**効果**: より詳細な進捗が見える

**実装工数**: 2-3時間（GrokClientの修正必要）

---

### 案C: システムプロンプトのキャッシュ（中優先度）

`buildSystemPrompt()` のDBアクセス結果をメモリキャッシュ。

```typescript
const promptCache = new Map<string, { prompt: string; expires: number }>();
```

**効果**: 50-200msの削減（全リクエストに影響）

**実装工数**: 1時間

---

### 案D: xAI接続の事前確立（検討段階）

Keep-Alive接続やPreconnectで、TCP/TLSハンドシェイクを削減。

**効果**: 不確実（xAI APIの仕様による）

**実装工数**: 不明（調査必要）

## 結論

**主要なボトルネックはxAI API側の「思考時間」（1-5秒）**であり、
これを短縮することは技術的に不可能。

**UX上の解決策は「待ち時間中のフィードバック」**を提供すること：
1. **スケルトン表示**で「動いている」ことを伝える
2. **ステータス詳細化**で「何をしているか」を伝える

これにより、ユーザーは「壊れたのではないか」という不安を感じず、
「処理中だから待っている」という理解ができる。
