# pdf-parse → unpdf 移行：ネイティブ依存パッケージの落とし穴

> **カテゴリ**: technology  
> **発生日**: 2026-03-21  
> **記録日**: 2026-03-22  
> **関連ファイル**: 
> - `lib/upload/file-parser.ts`
> - `types/pdf-parse.d.ts`（削除済み）
> - `tests/lib/upload/file-parser.test.ts`（新規）

---

## 概要（1行で）

PDF解析に `pdf-parse` を選定したが、ネイティブ依存（canvas）により WSL でインストールがハングし、Vercel デプロイでも動作しない構造だったため、ネイティブ依存ゼロの `unpdf` に切り替えた。

---

## 背景

ファイルアップロード機能で PDF 対応を追加する際、週間200万DLの `pdf-parse` を採用した。

---

## 問題

1. **npm install がハング**: WSL 環境で `npm install pdf-parse` が何度もタイムアウト。原因は `canvas` ネイティブモジュールのビルド（C++ ツールチェーン / node-gyp / Python 依存）
2. **Vercel デプロイ不可**: Vercel のサーバーレス環境には C++ ビルドツールがなく、`canvas` のコンパイルが失敗する。「pdf-parse kept breaking on Vercel」という報告が複数存在
3. **バンドラー問題**: webpack / Next.js が `pdf.worker.js` を解決できずエラーになるケースが多数
4. **暫定対応の負債**: 型定義ファイル手動作成、`eval` import での回避など、技術的負債が蓄積

---

## 解決策

`unpdf`（UnJS チーム製）に切り替え。

| 比較項目 | pdf-parse | unpdf |
|---------|----------|-------|
| ネイティブ依存 | canvas (C++ ビルド必要) | **ゼロ** |
| Vercel対応 | 動かない報告多数 | 公式対応 |
| Edge Runtime | 非対応 | 対応 |
| メンテナンス | 個人開発者 | UnJS (Nuxt/Nitro チーム) |
| npm install | WSL でハング | **3秒で完了** |

---

## 教訓

### 1. サーバーレス環境でのパッケージ選定は Vercel 互換性を最初に確認する

npm DL 数や API の使いやすさだけでなく、**デプロイ先の制約**（ネイティブモジュール不可、Worker ファイル解決等）を導入前に確認する。

### 2. ネイティブ依存のあるパッケージは原則避ける

`canvas`, `sharp`, `node-gyp` 依存のパッケージはサーバーレス環境で問題を起こしやすい。pure JS / WASM の代替があればそちらを優先する。

### 3. インストール困難は「環境の問題」ではなく「パッケージの問題」の可能性がある

「WSL のネットワークが遅い」と思い込んでいたが、実際はネイティブビルドがハングしていた。インストールに時間がかかる場合、パッケージの依存ツリーを先に確認すべき。

---

## 推奨事項

- PDF 解析には `unpdf` を使用する
- 新しいパッケージ導入時は `npm info <pkg> dependencies` でネイティブ依存を事前確認する
- Vercel デプロイ対象のプロジェクトでは、サーバーレス互換性を選定基準に含める
