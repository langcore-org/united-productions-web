## 概要

NextAuth.js + Neon から Supabase（認証+DB統合）に移行した際の知見。
本番環境構築を機に、技術スタックを見直し、統合エコシステムを選択した。

---

## 背景

### 当時の状況

- **開発環境**: Prisma ORM + Neon PostgreSQL + NextAuth.js
- **本番環境**: 未構築（新規作成が必要）
- **課題**: 
  - NextAuth.jsの複雑さ（コールバック処理、JWT管理）
  - Neon + NextAuth.jsの組み合わせでの管理の煩雑さ
  - 認証とDBが別システムによる運用オーバーヘッド

### 検討した選択肢

| 選択肢 | 選んだ理由 | 選ばなかった理由 |
|--------|-----------|-----------------|
| **パターンA**: Supabase完全移行 | 統合エコシステム、RLSポリシー、Realtime機能 | コード変更工数が大きい（88-132時間） |
| **パターンB**: Neon継続 | 既存コードの完全互換、最小限の変更 | 将来の拡張性に劣る、Supabase機能が使えない |
| **パターンC**: Supabase Auth + Prisma | 認証の簡素化、Prisma型安全性の維持 | Google Drive連携の複雑化、2つのクライアント管理 |

### 期待していた結果

- 認証フローの簡略化
- データベースと認証の一元管理
- RLSによるセキュリティ強化
- 管理画面による運用効率化

---

## 実際の結果

### 何が起こったか

**2026-03-20**: Supabase Auth + Supabase DBへの移行が完了。

1. **認証システム**: NextAuth.js → Supabase Auth
   - Google OAuth連携の簡略化
   - メール/パスワード認証の標準サポート
   - セッション管理の簡素化

2. **データベース**: Neon → Supabase PostgreSQL
   - Connection Pooler経由で接続
   - Prismaからの移行（部分的に維持）

3. **セキュリティ**: RLSポリシーの導入
   - ユーザーデータへのアクセス制御をDB層に集約
   - 認可ロジックの一元化

### 期待との差異

| 期待 | 現実 | 差異 |
|------|------|------|
| 完全なSupabase移行 | DBはSupabase、AuthもSupabase | 当初検討したPrisma維持パターンから変更 |
| 大規模なコード変更 | 比較的スムーズな移行 | 本番環境未構築だったため「新規構築」として実施 |
| Google Drive連携の複雑化 | 問題なく動作 | Supabase Authでも`provider_token`で取得可能 |

---

## 根本原因分析

### なぜ成功したのか

1. **「移行」ではなく「新規構築」**
   - 本番環境が未構築だったため、既存データの移行が不要
   - クリーンな設計からスタートできた

2. **段階的なアプローチ**
   - まずDBのみSupabaseに
   - Authも後からSupabaseに統合

3. **Prismaの部分的維持**
   - 型安全性を維持しつつ、Supabaseの機能も活用

### 判断ミスの要因（該当なし）

- [ ] 情報不足
- [ ] 検証不足
- [ ] 急ぎすぎた
- [ ] 過度な抽象化
- [ ] 将来予測の失敗
- [x] その他: 当初は移行コストを過大評価していたが、新規構築として実施したため問題なし

## 教訓

### 具体的な学び

1. **認証とDBを同じプラットフォームで統合すると運用が楽**
   - 管理画面が1つで済む
   - セッションとデータの連携がシンプル
   - バックアップ・監視が一元化

2. **RLSポリシーでセキュリティも確保しやすい**
   - アプリケーションコードでの認可チェックが減少
   - DB層で強制的にアクセス制御
   - セキュリティレビューが容易

3. **「新規構築」の機会を活かす**
   - 既存システムがある場合は移行コストが重い
   - 新規環境なら最新のベストプラクティスを採用可能

### 普遍化した原則

- **統合エコシステムの価値**: 異なるサービスの組み合わせより、統合されたプラットフォームの方が長期的にメリットが大きい場合がある
- **インフラ選定のタイミング**: 本番環境構築前は技術選定の最後のチャンス
- **段階的移行**: 全てを一度に変えるより、機能ごとに移行する方がリスクが低い

---

## 推奨事項

### 次回同様の状況でどうするか

```
1. 本番環境構築前に技術スタックを見直す
2. 認証+DB+Storageが統合されたプラットフォームを検討
3. 移行コスト vs 長期的運用コストを比較
4. 段階的な移行計画を立てる
5. RLSポリシーは初期から導入
```

### チェックリスト

新しいインフラを選定する前:

- [ ] 既存データの移行コストを見積もる
- [ ] 運用管理の工数を比較（ダッシュボード、バックアップ、監視）
- [ ] セキュリティ要件（RLS、認可）を確認
- [ ] 将来の拡張性（Realtime、Edge Functions等）を検討
- [ ] コスト比較（インフラ+運用工数）
