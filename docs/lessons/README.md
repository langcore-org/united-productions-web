## 記入すべきサイン

実装中に以下の思考が浮かんだら記入検討：

| 思考パターン | 例 |
|-------------|-----|
| 「あれ、思ってたんと違う」 | ライブラリが期待と違う機能しかない |
| 「結局、元に戻した」 | リファクタリング後、元の実装に戻す |
| 「3回目の修正だ…」 | 同じファイルに3回以上修正 |
| 「予定の2倍時間かかってる」 | 見積もり4時間が8時間超え |
| 「もっと早く気づけばよかった」 | 後からシンプルな解決策を発見 |

**判断基準**: 「3ヶ月後の自分が同じ状況で、これを知っていれば異なる判断をするか？」→ Yesなら記入

## 🔄 計画書から知見への変換ガイド

`plans/` で実装した計画から、技術的知見を抽出して `lessons/` に移動する手順。

### 変換のタイミング

実装完了直後、**記憶が新鮮なうちに**行う（推奨：完了当日〜3日以内）

### 変換手順

```bash
# 1. 計画書をベースに lessons ファイルを作成
cp docs/plans/your-plan.md docs/lessons/YYYY-MM-DD-title.md

# 2. 内容を書き換え（下記「変換ポイント」参照）
# 3. lessons/README.md に一覧追加
# 4. 元の計画書は削除（または archive/ に移動して参照先を記載）
```

### 変換ポイント

| 計画書の内容 | lessons への変換 | 理由 |
|-------------|-----------------|------|
| タスク一覧 | ✅ 結果サマリーに変更 | 「何をしたか」より「何が起きたか」 |
| 実装範囲 | ✅ 対象範囲として簡潔化 | 文脈を残す |
| 技術選定の理由 | ✅ 背景・判断基準として詳細化 | 次の選定の参考に |
| 問題と対応 | ✅ 「発見→調査→解決」ストーリーに | 再現性のため |
| 工数見積もり | ✅ 見積もり精度の分析に | 次回の改善に |
| スケジュール | ❌ 削除 | 知見として価値が低い |
| 期限・マイルストーン | ❌ 削除 | 知見として価値が低い |

### 必須セクション

変換後のドキュメントに必ず含める：

```markdown
## 概要
何についての知見か、1-2文で簡潔に

## 背景
- なぜその問題・技術に取り組んだか
- 前提条件・制約

## 結果
- 最終的に何を選んだ・どう解決したか
- コード例（該当する場合）

## 教訓
- 「もっと早く気づけばよかった」こと
- 試行錯誤した無駄なアプローチ

## 推奨事項
- 次に同じ状況になったときの推奨アクション
- 避けるべきアンチパターン
```

### 変換例

**計画書（ Before ）**:
```markdown
# API統合計画

## タスク
- [x] Grok APIクライアント実装
- [x] エラーハンドリング追加
- [ ] レート制限対応

## 期限
2026-02-20
```

**lessons（ After ）**:
```markdown
# Grok API統合の知見

## 概要
xAI Grok APIを直接呼び出す実装における設計判断と落とし穴

## 背景
LangChain経由だとツール呼び出しの制御が難しく、直接実装を検討

## 結果
- 直接実装を採用
- ストリーミングは SSE パーサーを自作

## 教訓
- LangChainの抽象化は今回のユースケースでは過剰
- 直接実装の方がテストしやすい

## 推奨事項
- ツール呼び出しを細かく制御したい場合は直接実装を検討
```
