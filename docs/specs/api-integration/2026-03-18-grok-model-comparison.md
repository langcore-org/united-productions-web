## 調査方法

同一クエリを両モデルに送信し、ストリームイベント・トークン・コスト・レスポンス品質を比較。

**クエリ**: 「テレビ番組「ザ・ノンフィクション」（フジテレビ）の最新放送回について、放送日・タイトル・内容を教えてください。」

**API**: `POST /responses` (xAI Responses API)、tools: `[web_search, x_search]`、stream: true

### cost_in_usd_ticks の換算式と検証

`cost_in_usd_ticks` はxAI APIのレスポンス `usage` に含まれる整数値。
**換算式: `cost_in_usd_ticks / 10,000,000,000`** でドル換算。

> xAI公式ドキュメントにこのフィールドの定義はないが、
> 複数パターンの API 実測で **公式単価 (token + tool) と完全一致** することを確認済み。
> **実際の課金額そのもの**と断定できる。

## 詳細分析

### 1. 応答速度

`grok-4.20-multi-agent-beta-latest` が約2倍速い (10秒 vs 23秒)。

**理由**: input_tokens_detailsによると、投入した191,203トークンのうち **139,776トークン (73%) がキャッシュ済み**。マルチエージェント内部で複数エージェントが同じ検索結果を共有することでキャッシュ効率が高く、実効処理が速い。

### 2. マルチエージェントの内部処理はストリームに出てこない

**重要**: `grok-4.20-multi-agent-beta-latest` のマルチエージェント動作はAPIストリームには露出しない。

ストリームイベントを完全キャプチャ (554イベント) した結果:
- ストリーム上のweb_search_callイベント: **4件** (並列2組で実行)
- usage の `num_server_side_tools_used`: **19件**
- 差分の15件 (= 内部エージェントの検索) はストリームに現れない

イベントタイプは両モデルで同一構造:
```
response.created
response.in_progress
response.output_item.added / done  <- web_search_call, message
response.web_search_call.in_progress / searching / completed
response.content_part.added / done
response.output_text.delta / done / annotation.added
response.completed
```

差異: `grok-4-1-fast-reasoning` のみ `response.custom_tool_call_input.delta/done` が出現 (x_search使用時)。

### 3. ツール呼び出しパターン

**grok-4-1-fast-reasoning**:
- web_search と x_search を両方使う
- ストリームに全ツール呼び出しが見える
- 逐次的: 1件ずつ実行されるケースがある

**grok-4.20-multi-agent-beta-latest**:
- web_search のみ使用 (x_searchは内部的に呼ばない)
- ストリーム上は並列2件ずつ: `_0/_1` -> 完了 -> `_2/_3` -> 完了
- 残り15件の検索は内部マルチエージェント層で処理され不可視

### 4. トークン消費とキャッシュ

`grok-4.20-multi-agent-beta-latest` のinputトークンが約6倍多い (170K〜222K vs 30K)。これはマルチエージェントが内部で複数回の推論・検索サイクルを回すため。

xAI APIサーバー側が自動でキャッシュしており（こちらのコードに実装はない）、大部分のトークンが割引扱いになる。`reasoning_tokens: 5,918` が発生しており、内部でchain-of-thought推論を実行している。

### 5. レスポンス品質

**grok-4-1-fast-reasoning** の出力:
```
**最新放送回（2026年3月17日現在）：**
- **放送日**: 2026年3月15日（日）14:00〜
- **タイトル**: 家庭教師のマリアさん 〜女装と恋と推し活と〜
- **内容**: ピンクの服で女装する家庭教師...（簡潔な要約）
```
-> 箇条書き形式。次回放送情報も含む。簡潔でコンパクト。

**grok-4.20-multi-agent-beta-latest** の出力:
```
**2026年3月15日(日) 14:00〜14:55放送の「家庭教師のマリアさん...」**

### タイトル
### 内容概要
- 主人公の生き方
- 経歴（詳細な経緯まで記述）
- 人間関係
- 夢
```
-> 見出し付き構造化形式。主人公の経歴・動機まで詳細に記述。より情報量が多い。

## 技術メモ

- 両モデルともAPIストリームのイベント構造は同じ
- マルチエージェント動作はサーバーサイドで完結し、クライアントには集約された結果のみ届く
- `usage.server_side_tool_usage_details` でモデルが内部で何回検索したか確認可能
- `usage.input_tokens_details.cached_tokens` でキャッシュ効率を確認可能
- `usage.cost_in_usd_ticks` で実際の課金額を取得可能（**÷ 10,000,000,000 でドル換算**）
- billing/usage API エンドポイントは存在しない（`/usage`, `/billing`, `/organization` 等すべて404）
- 調査スクリプト: `scripts/compare-grok-models.ts`, `scripts/compare-grok-raw-events.ts`, `scripts/verify-cost-formula.ts`, `scripts/verify-cost-billing.ts`

### 補足・制約事項

1. **ラインナップ情報**: 現在一時停止中（2026-03-19）。含める場合は +1,000〜3,000 tokens 増加
2. **キャッシュ**: xAI APIはサーバーサイドで自動キャッシュ。同一プロンプトは `cached_input` 扱いで割引適用
3. **トークナイザー**: cl100k_base（GPT-4/Claude/Grok共通）

### 所見

現状の **~4,700 tokens** は Grok/GPT-4 のコンテキストウィンドウ（128K〜2M）に対して約 **0.2〜3.7%** の占有率であり、技術的には問題なく収まっている。

ただし以下の点を考慮すべき：
