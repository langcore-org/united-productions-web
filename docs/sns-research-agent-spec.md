# SNS人物探索エージェント仕様書

## 質問リスト（回答待ち）

以下の質問に回答してください。ドキュメント内の該当箇所に直接記入してください。

### Q1: 表示形式の選択理由
セクション 2.3 で「視覚的（絵文字、区切り線等）」を選択しましたが、その理由は何ですか？

**推奨**: 「AIエージェントの進捗を直感的に把握しやすく、ユーザー体験を向上させるため」
**判断材料**: manus/genspark/claude code等の人気AIツールは全て視覚的な進捗表示を採用しており、ユーザーの期待に合わせる必要がある

### Q2: 詳細調査のレベル
セクション 3.3 の調査レベル（ライト/スタンダード/ディープ）から、どれを選択しますか？

**推奨**: 「スタンダード」
**判断材料**: 
- ライト: 情報不足で候補者の質がわからない
- スタンダード: 投稿分析まで含めて適切な評価が可能
- ディープ: 時間がかかりすぎ、リアルタイム性が失われる

### Q3: タイムアウト時間
セクション 4.3 の人間介入時、何秒で自動続行しますか？

**推奨**: 「60秒」
**判断材料**: 
- 短すぎる（10-30秒）: ユーザーが読む時間がない
- 長すぎる（5分以上）: ワークフローが停滞する
- 60秒: 候補者リストを確認し、判断するのに十分な時間

### Q4: 実行環境
セクション 8.1 から実行環境を選択してください：
- [x] LangGraph Cloud（マネージド）
- [ ] 自前インフラ（Vercel + 別途サーバー）
- [ ] ハイブリッド（LangGraph + Inngest for スケジュール）

**確定**: 「LangGraph Cloud（マネージド）」
**理由**: 無料枠で十分に検証可能。開発速度優先でインフラ管理不要。

### Q5: APIキー管理
セクション 8.2 APIキーは誰が持ちますか？
- ユーザー持ち / システム持ち / 両方

**推奨**: 「システム持ち（管理者が管理）」
**判断材料**: 
- ユーザー持ち: 設定が複雑になり離脱率が上がる
- システム持ち: すぐに使い始められ、体験がスムーズ
- 両方: 運用が複雑

### Q6: 候補者上限
セクション 8.2 1回の探索あたりの候補者上限は何人ですか？

**推奨**: 「20人」
**判断材料**: 
- 多すぎる（50人以上）: 情報過多、選択疲れ
- 少なすぎる（5人以下）: 選択肢が少ない
- 20人→絞り込み→5人: 適切なバランス

### Q7: 実行回数上限
セクション 8.2 1日の実行回数上限は何回ですか？

**推奨**: 「10回/日（無料）/ 無制限（有料）」
**判断材料**: 
- 無制限: APIコストが予測不能
- 厳しすぎる（1-2回）: 使い物にならない
- 10回: 通常の探索ニーズを満たし、濫用を防ぐ

### Q8: 探索履歴の保存
セクション 8.3 探索履歴を保存しますか？
- する / しない

**推奨**: 「する」
**判断材料**: 
- 保存しない: 過去の結果を参照できない、再利用不可
- 保存する: 再探索の基準にできる、進捗管理が可能

### Q9: 保存期間
セクション 8.3 保存する場合、何日間保存しますか？

**確定**: 「無制限（または技術的な限界まで）」
**理由**: 長期的なトレンド分析や過去の候補者との比較に必要。ストレージコストは後で最適化。

### Q10: 個人情報の取り扱い
セクション 8.3 候補者の個人情報をどのように取り扱いますか？

**推奨**: 「探索履歴として保存するが、第三者への提供はしない。レポート出力時はユーザー責任とする」
**判断材料**: 
- SNS上の公開情報を収集するだけなので法的問題は少ない
- ただし、連絡先等の取得には注意が必要
- 利用規約で免責事項を明記

## 概要

LangGraphを使用したSNS人物探索エージェントの仕様書。
リアルタイムで実行状況をチャットに表示しながら、SNS上のインフルエンサー・専門家を発見・評価する。

**対象ユースケース**: 番組出演者候補探索（例: 片付けの専門家でYouTube人気・関東圏在住）

---

## 1. モード定義

各モードは独立したワークフローを持ち、サイドバーから選択して実行する。

### 1.1 人物探索モード（本仕様の主軸）

**目的**: 特定分野の専門家・インフルエンサーを発見・評価

**入力例**:
- 「片付けの専門家で、YouTubeで人気があり、関東圏在住の人」
- 「フィットネストレーナーで、TikTokで若年層に人気の人」
- 「料理研究家で、Instagramでビジュアルが綺麗な人」

**出力**: 候補者リスト（プロフィール・評価スコア付き）

### 1.2 他モード（将来追加予定）

- トレンド分析モード
- 競合分析モード
- キャンペーン効果測定モード

## 3. ワークフロー制御仕様

### 3.1 候補者発見フロー

```
[キーワード入力] 
    ↓
[プラットフォーム選択] → YouTube / X / Instagram / TikTok
    ↓
[検索・候補者リストアップ] → 最大___人まで
    ↓
[基本フィルタ適用] → 登録者数 / フォロワー数 / 投稿頻度
    ↓
[詳細プロフィール収集] → 経歴 / 実績 / 地域 / 連絡先
    ↓
[評価スコアリング] → 各候補を多軸で評価
    ↓
[絞り込み・ランキング] → トップ___人を選定
    ↓
[レポート生成]
```

### 3.2 評価基準（スコアリング）

**方針**: ユーザーのプロンプトに応じて動的に決定。デフォルト値は設定しない。

**評価軸の候補**（必要に応じて選択・追加）:
- 人気度: 登録者/フォロワー数、エンゲージメント率、成長率
- 専門性: コンテンツの特化度、資格・認定、経験年数
- 実績: メディア出演、出版、受賞歴、講演実績
- キャラクター: 話しやすさ、ユーモア、信頼感（動画分析）
- 地域適合: 指定地域への居住/活動実績
- その他: ユーザー指定のカスタム軸

**重みの決定方法**: ユーザーの入力文からxAI Grokが自動抽出・提案し、ユーザー確認後確定

### 3.3 詳細調査の深さ

| 調査レベル | 収集情報 | 所要時間 | 対象 |
|-----------|---------|---------|------|
| **ライト** | SNSプロフィール基本情報 | 短 | 全候補者 |
| **スタンダード** | + 最近投稿分析 + エンゲージメント詳細 | 中 | 上位___人 |
| **ディープ** | + 経歴詳細 + メディア実績 + 評判調査 | 長 | 最終候補___人 |

**確定**: スタンダード

**詳細**:
- 最初の候補者リストアップ: 20人
- スタンダード調査対象: 上位10人
- 最終選定: 5人

### 3.4 絞り込み条件

**方針**: ユーザーのプロンプトに応じて動的に決定。デフォルト値は設定しない。

**絞り込み条件の候補**（必要に応じて選択）:
| 条件 | 例 |
|------|-----|
| 最低登録者/フォロワー数 | 10万人以上 |
| 最低投稿頻度 | 月4回以上 |
| 地域 | 関東圏、東京都、etc |
| 経験年数 | 3年以上 |
| メディア出演経験 | あり優遇、必須、etc |
| 性別 | 男性、女性、どちらでも |
| 年齢層 | 20代〜40代 |
| その他 | ユーザー指定条件 |

**決定方法**: ユーザーの入力文からxAI Grokが自動抽出・提案し、ユーザー確認後確定

### 5.3 APIリスクと対応方針

**基本方針**: 公式APIにリスク（高額・審査・制限）がある場合は、**別ツールへの切り替えを検討**

| プラットフォーム | 現状のリスク | 推奨対応 |
|----------------|------------|---------|
| **YouTube** | 低リスク（無料・安定） | Data APIを継続使用 |
| **X** | 高リスク（$200/月〜・制限厳しい） | **代替ツール検討**: RapidAPIのX API、または探索対象から除外 |
| **Instagram** | 中リスク（Businessアカウント必須・個人アカウント不可） | **代替ツール検討**: Apify Instagram Scraper、またはWeb検索で補完 |
| **TikTok** | 高リスク（Research API審査長期化・商用不可） | **代替ツール検討**: TikTok Creative Center（手動参照）、サードパーティAPI、または探索対象から除外 |

**推奨する初期構成**:
- MVP: **YouTubeのみ**（最も信頼性高い）
- 段階的に追加: Web検索 → Apify等の代替ツール → 公式API（審査通過後）

interface PersonDiscoveryState {
