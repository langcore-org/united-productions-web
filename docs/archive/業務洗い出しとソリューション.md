# 業務洗い出しとソリューション

**プロジェクト:** UNITED PRODUCTIONS様 AI導入・DX化
**作成日:** 2026年2月12日
**参照元:** 2026年2月5日ミーティング（書き起こし1〜3、galaxy書き起こし、要約・議事録）。United Productions様 LangCore 共有用フォルダ内の成果物は**例として提供されたものであり、実際の管理・運用を示すものではない**。

---

## ソリューション種別の定義

| 種別 | 説明 |
|------|------|
| **ツール導入** | 既存SaaS・AIサービス・Google Workspace機能などの導入 |
| **LC開発** | 社内ウェブアプリや簡易ツールをLCが開発 |
| **運用・その他** | ルール整備、運用改善、将来的な検討事項 |

**ソリューションの考え方（現状の資料・会議方針に基づく）**  
まず**ツール導入**（Grok・Perplexity・Zoom文字起こし・Gemini for Workspace等の既存サービス活用）と**運用・ルール整備**（ドライブ集約ルール、議事録フォーマット、マニュアル等）で効果を出し、効果が確認できた領域で**LC開発**（チャットボット・検索ポータル・台本ツール等）を発展形として検討する。議事録・文字起こしはZoom主流の現状を前提とし、起こし・NA原稿は既存ツール（Premiere内蔵AI・Notta）を優先し、不足時はLCモック検討とする。

## フェーズ1：企画〜リサーチ

---

### 1. 専門家（マニア）候補のリサーチ（優先度：高）

**現状・課題**
- リサーチ会社（ビスポ社）への外注や、ADによる手動のSNS・YouTube検索が主
- 候補者30〜40人をリストアップし、Zoom面談で絞り込む
- 膨大なSNS情報から候補者を探す工数が非常に重い
- AIに慣れていないADは「何をどう聞けばいいか」がわからず活用できない

**ソリューション**

| # | ソリューション | 種別 |
|---|--------------|------|
| 1-a | **Grok・Perplexity・社内ドライブを参照するチャットアプリの構築** — 業務1・2・20を一括で対応。Grok API（SNSリアルタイム検索）・Perplexity API（ソース付きWeb検索）・社内ドライブ（Google Drive API等で過去リサーチ資料・ロケ地情報を横断参照）を統合したチャットUIをLCが構築。専門家候補のリスト生成・ロケ地ナレッジ検索・考査用エビデンス確認を同一アプリで行う | LC開発 |
| 1-b | **Grok（X Premium）の直接利用（暫定）** — チャットアプリ構築までの暫定として、Grokを社内で利用可能にし、候補者リストを一度に生成して人間が選別する運用 | ツール導入 |
| 1-c | **リサーチ手順マニュアルの整備** — AIに慣れていないAD向けに、プロンプト例や検索のコツを番組ジャンル別にまとめた社内ガイドを作成し周知する | 運用・その他 |

### 2. ロケ場所のリサーチ（優先度：高）

**現状・課題**
- Space Market・SHOOTEST・不動産会社など、複数のサイトや業者に個別問い合わせしており煩雑
- 条件（一軒家・アイランドキッチン・3LDK以上・都内・予算内・長期貸し出し可など）が多岐にわたる
- 過去に使ったロケ地の情報が各番組チームに閉じており、社内で共有されていない
- 「テレビ撮影に適した場所」は一般的なGoogle検索ではヒットしにくい業界特有の情報

**ソリューション**

| # | ソリューション | 種別 |
|---|--------------|------|
| 2-a | **Grok・Perplexity・社内ドライブを参照するチャットアプリで対応** — 業務1-aのチャットアプリにロケ地・場所系の検索を統合。社内ドライブに集約した過去のロケ地資料を参照し、条件（エリア・用途・予算帯）や「過去に使った場所」「面倒な店主」等のナレッジを横断検索できるようにする | LC開発 |
| 2-b | **ドライブ集約ルール策定** — チャットアプリが参照するドライブ資料を整えるため、「日常業務で作った資料をドライブに入れる」ルールを策定。LCからフォルダ構成案を提示し、UPから号令を出す | 運用・その他 |
| 2-c | **NotebookLMによる簡易ナレッジ検索（暫定）** — チャットアプリ構築前の暫定として、ドライブに集約した資料をNotebookLMで横断検索する運用を開始する | ツール導入 |

### 6. 議事録・文字起こし（会議・面談）（優先度：最高）

**現状・課題**
- 会議中に手打ちでタイピングして記録するADもいれば、録画を見直して起こす人もおり、方法がバラバラ
- これまでZoom/Teamsの標準文字起こしは精度が低いとされていたが、**Zoomの文字起こしを試した結果、十分使えそう**（さらなるテスト後に本格利用を検討）
- 「テキスト化」（音→テキスト）が課題だったが、Zoom文字起こし採用後は**その後のフォーマット成形**（議事録形式・話者分離・要約）のソリューション検討が次の焦点
- 業界慣習でZoomが主流、対外的にGoogle Meetへの切り替えが難しい
- マニア候補とのZoom面談（20人以上）の議事録作成にも多大な工数がかかっている

**ソリューション**

| # | ソリューション | 種別 |
|---|--------------|------|
| 6-a | **Zoom文字起こしの本格利用** — 会議で「十分使えそう」との評価あり。さらなるテスト（会議種別・話者数・雑音環境など）を実施したうえで、Zoom標準の文字起こしを議事録作成の第一段階として利用する。対外・社内問わずZoom主体の現状に合致する | ツール導入 |
| 6-b | **文字起こし後のフォーマット成形** — Zoom文字起こしテキストを、議事録フォーマット（見出し・議題・決定事項・TODO）や面談用フォーマット（人物名・経歴・話した内容・出演可否）に整形する。まずはGemini / Claude に貼り付けて後処理する運用から開始。効果が出ればLC開発の簡易ツール（アップロード→整形出力）を検討 | ツール導入 / LC開発 |
| 6-c | **Otter.ai / Notta等の補助** — Zoom文字起こしで不足する場合の補助として、高精度な有料文字起こしツールを検討。社内会議に限りGoogle Meet化する場合はWorkspace議事録も選択肢 | ツール導入 |
| 6-d | **議事録運用ルールの統一** — 「Zoom文字起こし→どのフォーマットで整形するか」「どこに保存するか」を全番組共通で策定する | 運用・その他 |

## フェーズ2：構成資料〜ロケ準備

---

### 7. 構成資料の作成（優先度：中）

**現状・課題**
- 写真中心で「今回こんなことをやりたい」を初見の人に伝える資料。会議スライド代わりにもなる
- 写真選びが命で、テキストよりビジュアル重視
- Geminiに「写真付きでポップな資料を」と指示したが、フォント・レイアウトが期待に届かなかった
- 文字情報（見出し・制約情報・注意点など）を調べて書く部分に最も時間がかかる
- リサーチ→構成資料→会議→再リサーチのループ構造で、何度も更新が発生する

**ソリューション**

| # | ソリューション | 種別 |
|---|--------------|------|
| 7-a | **見出し・テキスト部分のみAI生成** — 写真付きレイアウトは期待に届きにくいため、見出し（太字）・説明文・制約情報のみGemini / Claude で叩き台を生成する。写真の選定・配置は人間が担当する使い分けを定着させる | ツール導入 |
| 7-b | **構成資料テンプレートの標準化** — Googleドキュメント or Wordで、写真スペース・見出し・注意事項枠が事前に配置された共通テンプレートを作成し、各番組で利用する | 運用・その他 |

---

### 8. ロケ台本の作成（優先度：中）

**現状・課題**
- 構成資料を1日のスケジュール単位で詳細化した台本（セリフ案・ナレーション原稿・進行順等）
- 第一弾を出してから会議でフィードバック→修正を繰り返してブラッシュアップ
- Gemini→Wordへのエクスポートでフォント・レイアウトがズレる
- 会話の流れが入るWord独自の箱型フォーマットがあり、AIでの自動生成が難しい

**ソリューション**

| # | ソリューション | 種別 |
|---|--------------|------|
| 8-a | **AI叩き台 + 手動貼り付けの運用** — Gemini / Claude にプロット（テーマ・場所・出演者・やりたいこと）を入力し、テキストのみ生成。既存のWordテンプレートに手動で貼り付ける運用を標準とする。Word出力のフォント・レイアウトずれを避けつつ工数削減 | ツール導入 |
| 8-b | **ロケ台本自動生成ツールの構築（発展形）** — 叩き台＋手動運用で効果が出た場合に検討。プロット入力→既存Wordフォーマットにセリフ案・ナレーション案・進行順を流し込むウェブアプリ（Claude/Gemini API + python-docx）。LCがモックを試作し有効ならスコープに組み込む | LC開発 |

### 14. 起こし・NA原稿の作成（優先度：最高）

**現状・課題**
- 映像チェック用に、喋り部分を文字起こし + ナレーション（N）をWord化する
- Premiereのテキスト抽出機能でタイムコード付きテキストを出力→Wordで整形する人と、手打ちの人が混在
- 従来はPremiereの文字起こし精度に不満がある声が多かったが、**2026年現在、Premiere ProのAI（Textパネル → 文字起こし）が大幅進化**という口コミあり。内蔵AIの再評価が有効
- Word出力が必要な場合は**Notta**などWordエクスポート対応の文字起こしツールも選択肢
- 話者分け（「櫻井」「末澤」「N」等）を頭につけた起こしが自動生成されれば理想的
- 台本と同一のWordフォーマットで出力する必要がある
- 会議で「ADとして一番楽になるのは議事録と文字起こし」と明言されている

**ソリューション**

| # | ソリューション | 種別 |
|---|--------------|------|
| 14-a | **Premiere Pro内蔵AI文字起こしの活用** — Textパネル → 文字起こしをまず試す。2026年時点でAIが大幅進化しているとの評価があり、既存ワークフロー（Premiere内で完結）で精度が足りればコスト最小。抽出→Word整形の標準手順を策定する | ツール導入 |
| 14-b | **NottaによるWord出力** — Word形式での起こし原稿出力が必要な場合、NottaのWordエクスポート機能を利用する。文字起こし→話者分離→Word出力の一連フローが可能 | ツール導入 |
| 14-c | **起こし・NA原稿自動生成ツールのモック検討** — 内蔵AI・Nottaで不足する場合の選択肢として、LCが音声→文字起こし→固定Wordレイアウト＋話者分け（櫻井・末澤・N等）のモックを検討。有効ならスコープに組み込む | LC開発 |
| 14-d | **Premiere文字起こしの精度検証と運用ルール** — 最新版のTextパネル文字起こしを実素材で検証し、「抽出→整形」の標準手順とフォーマットを社内で共有する | 運用・その他 |

### 15. テロップ原稿の作成（優先度：低）

**現状・課題**
- タイムコードごとに「何秒に何を入れるか」「左縦書き / 右縦」等を紙上で指示する資料
- 編集所のプロがデザインして挿入。音テロ（SE付き）は音効に別発注
- ディレクターによってPremiere上に仮テロップを入れる人と、ドキュメントだけ渡す人がいる

**ソリューション**

| # | ソリューション | 種別 |
|---|--------------|------|
| 15-a | **起こし原稿からのテロップ原稿自動生成** — 業務14-aで生成した起こし原稿から、AIが「テロップにすべき箇所」を推定し、タイムコード・テロップ内容・配置指示の叩き台をExcelで出力する。最終的な取捨選択はディレクターが判断 | LC開発 |
| 15-b | **現行運用の継続（当面）** — テロップの選定はディレクターのセンスに依存する部分が大きいため、当面は現行運用を維持しつつ、14-aの起こし精度向上を優先する | 運用・その他 |

### 16. カンペの作成（優先度：低）

**現状・課題**
- ロケ台本から演者（櫻井翔等）のコメント部分のみを抽出し、スケッチブック等の大きな紙に印刷

### 20. 情報エビデンス確認・考査対応（優先度：高）

**現状・課題**

- 薬機法・差別用語・表現の妥当性チェックも含む
- 一般人のブログ等は出典として使えず、信頼できるソースが必要

**ソリューション**

| # | ソリューション | 種別 |

| 20-b | **Perplexity Pro・Grokの直接利用（暫定）** — チャットアプリ構築までの暫定として、Perplexity Pro・Grokを社内導入し、エビデンス確認・考査の補助に利用する | ツール導入 |
| 20-c | **考査チェックリスト付きウェブツール（発展形）** — 薬機法NG表現・差別用語・過去の考査指摘をDB化し、原稿テキストでリスク箇所を検出するウェブツール。チャットアプリに統合または別途構築 | LC開発 |

### 26. 社内ナレッジ共有基盤の構築（優先度：最高）

**現状・課題**
- 番組ごとにBOX・Dropbox・Googleドライブなどバラバラなクラウドストレージを使用
- 過去のリサーチ資料・ロケ地情報・ノウハウが各チームに閉じている
- 「この場所、前に使ったことある？」「この店主は面倒くさい」等の情報が個人の記憶に留まっている
- BOXは容量無制限で映像保管に適しているが、AI連携には不向き

**ソリューション**

| # | ソリューション | 種別 |
|---|--------------|------|
| 26-a | **Googleドライブへのナレッジ集約ルール策定** — リサーチ資料・ロケ地情報・議事録等を、LCが提案するフォルダ構成に従ってGoogleドライブに集約するルールを策定する。まず「日常業務で作った資料をドライブに入れる」ルールから開始。映像データはBOX継続でも、テキスト系資料はドライブに統一 | 運用・その他 |
| 26-b | **NotebookLMによるナレッジ横断検索** — ドライブに集約した資料をNotebookLMのソースとして登録し、「過去にカラオケで使った場所は？」等の自然言語で横断検索できるようにする。アプリ構築前の現実的な第一歩として有効 | ツール導入 |
| 26-c | **社内ナレッジ検索ポータルの構築（発展形）** — 集約・NotebookLM運用で効果が出た場合に検討。Vertex AI Search等でチャット形式の検索ポータルを構築する | LC開発 |
