| 機能 | 理由・方針 |
|------|------------|

## 2. 機能別実装プラン

### 2.1 リサーチ機能（出演者・エビデンス）

#### 方針

- **出演者リサーチ**: 現状のX検索＋Web検索を軸に、プロンプトの磨き込みを重点的に実施。
- **エビデンスリサーチ**: 出演者リサーチに近い形（参照・出典を明示する出力形式）で実装し、3月で届く範囲に収める。
- **SNS**: Xを軸とする。YouTube・Instagramは外部有料ツールの統合を「近いうちに」検証。TikTokはAPI申請に時間がかかるため後回し。
- リサーチ機能内の各要素（出演者／エビデンス／場所／情報）は、それぞれ別開発コースとして扱う前提。

#### タスク一覧

- [ ] 出演者リサーチ用プロンプトの磨き込み（出力形式・参照の出し方・番組情報の参照方法）
- [ ] エビデンスリサーチ用プロンプト・出力形式の整備（参照・出典を明確に）
- [ ] 番組情報（レギュラー番組一覧等）の正確性検証とデータ修正・拡充
- [ ] 思考プロセス等の不要な表示を非表示にする（UI調整）
- [ ] （時間あれば）YouTube/Instagram用有料ツールの統合検証 → 結果を backlog に記録

#### 参照

- 会議メモ: リサーチ結果の正確性確認（島田指摘）、X＋Web検索の説明、プロンプト磨き込みと2機能優先の合意。
- 既存仕様: [specs/features/research-cast.md](../specs/features/research-cast.md), [research-evidence.md](../specs/features/research-evidence.md)

### 2.2 議事録作成

#### 方針

- **入力**: Zoom から出力される **VTT または TXT ファイルのアップロード** を基本とする（佐久間合意）。
- 初期画面を「まずファイルをアップロードしてください」とする形に変更する。

#### タスク一覧

- [ ] 議事録作成の初期画面を「ファイルアップロード基本」に変更（VTT/TXT対応を明示）
- [ ] ファイルアップロードから文字起こしテキスト取得までのフロー実装・動作確認
- [ ] 既存の議事録生成API・プロンプトとの結合確認
- [ ] ユーザー向けガイド・ラベル文言の整備（「ZoomのVTT/TXTをアップロード」等）

#### 参照

- 会議メモ: 議事録はファイルアップロードが最も効率的という見解、初手画面をファイルアップロードにすることの合意。
- 既存仕様: [specs/features/minutes.md](../specs/features/minutes.md)

### 2.3 新規企画立案

#### 方針

- **名称**: 「新規企画立案」で統一。サブテキスト例:「アイデアを出してくれます」（3月後半にデザイン含め最終調整）。
- **実装**: 過去に作成した対話型の新規企画立案用プロンプトを、**過去番組情報の事前インプットなし**で現アプリに移植し、動作確認する。
- ユーザーがデータ（PDF等）をアップロードして企画を助ける形は「まずはそれでよい」という合意。ファイルアップロード対応は実装状況に応じて（最悪なくても、番組名などをユーザーが打ち込む形で代替可能）。
- 公開されている番組情報の事前インプットは並行して拡充。社内の過去企画情報のインプットは権限切り分け・ミス防止の実装が重いため、現実的には余裕があれば対応。

#### タスク一覧

- [ ] 過去アプリの新規企画立案用プロンプトを現アプリの「新規企画立案」エージェントに移植
- [ ] 過去情報なしでの動作確認（対話の流れ・出力の確認）
- [ ] サイドバー等の表示名を「新規企画立案」に統一し、サブテキスト「アイデアを出してくれます」を表示（仮でよい）
- [ ] （任意）番組情報の事前インプット拡充（公開情報ベース）。新規企画立案以外の機能でも共有利用。
- [ ] 3月後半: デザイン含む名称・サブテキストの最終調整

#### 参照

- 会議メモ: 移植可能性の確認、過去情報なしで移植して動作を見る合意、ネーミング議論と「新規企画立案」＋サブテキストで一旦決定。
- 既存仕様: [specs/features/proposal.md](../specs/features/proposal.md)

### 2.4 ナレーション原稿作成（優先度下げ・後続）

#### 方針

- 第1段階リリースでは**含めない**。他機能の品質・リリースを優先。
- Word 内の表（タイムコード・セリフ・発言者）への変換は実装が重く、崩れが出ると作業が無意味になるリスクあり。時間をかけてフルセット（画面上での編集～出力まで）を実装するまでは優先度を下げる。
- 将来やる場合のイメージ: 画面上で編集可能にし、完成に近い形まで持っていってから Word/表形式で出力するフルセットを検討。

#### 後続タスク（backlog で管理）

- [ ] Word 表形式の仕様確定と実装可否の再検証
- [ ] 表のテキストコピー・ペースト等の中間出力で価値が出るか検証
- [ ] 画面上編集＋出力までのフルセット設計

#### 参照

- 会議メモ: Word表の難しさ、中途半端な完成度でADが離れる懸念、佐久間・鵜飼・浅野の優先度下げ合意。

### 2.5 認証・テスト環境

#### 方針

- 現状はテスト段階。Google 管理画面でテストユーザーとして登録したアカウントのみログイン可能。
- 希望者にはテストユーザー登録を案内して試用可能にする。
- テストから本番への移行は「近いうちに優先度を見て実施」する。

#### タスク一覧

- [ ] テストユーザー登録手順の明文化・共有
- [ ] 本番移行のタイミング決定と実施（リリース目標に合わせて）

---

### 2.6 デザイン・UI 調整

#### 方針

- 3月12日の現場テスト、および 3月後半 に対面で相談し、最終調整する。
- 島田要望: 左上のUPロゴ、Teddy下のキャプション（例: Teacher & Buddy / AD Production 等）、ロゴ・アイコンの見栄え。
- ロゴ・アイコンは指定があれば実装に反映可能。ログイン前画面のプレースホルダー（クマ等）も共通ロゴに差し替え可能。

#### タスク一覧

- [ ] 3/12 現場テストまで: 現状のままで利用可能であることを確認
- [ ] 3月後半: ロゴ・Teddy下キャプション・新企画立案のサブテキスト等の最終調整を反映
- [ ] 社内発表用: **ナレーション原稿を省いた画面**のスクリーンショットを島田に共有（例: チャット画面で空で可）

#### 参照

- 会議メモ: デザイン・ロゴの要望、ロゴ指定があれば反映可能、3月後半に最終調整の合意、スクショはN原稿を省いたもの。
