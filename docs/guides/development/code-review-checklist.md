# コードレビューチェックリスト

> **プルリクエスト時のレビュー基準**
> 
> **最終更新**: 2026-02-20 13:16

---

## ✅ 全般チェック

### 型安全性
- [ ] `any` 型の使用が適切か（避けるべきか、正当化されているか）
- [ ] 新規型定義は `types/` ディレクトリに配置されているか
- [ ] 既存の型定義を再利用・拡張しているか
- [ ] ジェネリクスの使用が適切か

### エラーハンドリング
- [ ] `try-catch` で適切にエラーを捕捉しているか
- [ ] カスタムエラークラス（`AppError`等）を使用しているか
- [ ] エラーメッセージはユーザーにとって理解しやすいか
- [ ] エラーログは適切なレベル（INFO/WARN/ERROR）で記録されているか

### ログ記録
- [ ] デバッグ用の `console.log` が残っていないか
- [ ] 本番環境で必要なログは `lib/logger.ts` を使用しているか
- [ ] 機密情報（パスワード、APIキー等）がログに出力されていないか

---

## 🎨 フロントエンド

### Server/Client 境界
- [ ] Server Component でブラウザAPI（`localStorage`, `window`等）を使用していないか
- [ ] Client Component には `"use client"` ディレクティブが付いているか
- [ ] データ取得は Server Component で行い、propsで渡しているか

### React ベストプラクティス
- [ ] 不要な `useEffect` / `useState` がないか
- [ ] 依存配列が適切に設定されているか
- [ ] メモ化（`useMemo`, `useCallback`）が必要な箇所で使用されているか
- [ ] 無限ループのリスクがないか

### アクセシビリティ
- [ ] 画像には適切な `alt` 属性があるか
- [ ] フォーム要素には `label` が紐づいているか
- [ ] キーボード操作が可能か
- [ ] コントラスト比が十分か（WCAG 2.1 AA基準）

### UI/UX
- [ ] ローディング状態の表示があるか
- [ ] エラー時のフォールバックUIがあるか
- [ ] レスポンシブ対応が考慮されているか
- [ ] アニメーションは適度でパフォーマンスに影響しないか

---

## 🔌 API

### 入力バリデーション
- [ ] Zodスキーマで入力を検証しているか
- [ ] バリデーションエラーは適切なレスポンスを返すか
- [ ] ファイルアップロードのサイズ制限があるか

### 認証・認可
- [ ] 保護されたエンドポイントに認証チェックがあるか
- [ ] ユーザー権限の確認が必要な場合は実装されているか
- [ ] レート制限が適用されているか

### レスポンス
- [ ] エラーレスポンスは統一された形式か
- [ ] HTTPステータスコードは適切か
- [ ] レスポンスに機密情報が含まれていないか

---

## 🗄️ データベース

### Prisma
- [ ] クエリは効率的か（N+1問題がないか）
- [ ] トランザクションが必要な箇所で使用されているか
- [ ] マイグレーションファイルが生成されているか

### スキーマ変更
- [ ] 破壊的変更を避けているか（または移行戦略があるか）
- [ ] インデックスが適切に設定されているか
- [ ] 新規カラムにはデフォルト値が設定されているか

---

## ⚡ パフォーマンス

### フロントエンド
- [ ] 不要な再レンダリングがないか
- [ ] 大きなリストは仮想化されているか
- [ ] 画像は最適化されているか（`next/image`の使用）

### API
- [ ] 重い処理は非同期で行われているか
- [ ] キャッシュ戦略が適切か
- [ ] データベースクエリは最適化されているか

### バンドルサイズ
- [ ] 新規パッケージの追加は必要最小限か
- [ ] 動的インポートでコード分割が行われているか

---

## 🧪 テスト

### 単体テスト
- [ ] 新規ユーティリティ関数にはテストがあるか
- [ ] エッジケースがカバーされているか
- [ ] テスト名は明確で意図が伝わるか

### 統合テスト
- [ ] APIエンドポイントの主要フローがテストされているか
- [ ] エラーケースもテストされているか

### E2Eテスト
- [ ] クリティカルパスがカバーされているか
- [ ] 変更した機能に関連するテストが更新されているか

---

## 📝 コード品質

### 命名
- [ ] 変数・関数名は意図が明確か
- [ ]  boolean変数は `is`/`has`/`can`/`should` プレフィックスを使用しているか
- [ ] コンポーネント名は機能を表しているか

### コメント
- [ ] 複雑なロジックには説明コメントがあるか
- [ ] TODOコメントには理由と対応期限が記載されているか
- [ ] 不要なコメントアウトコードが残っていないか

### 一貫性
- [ ] 既存のコードスタイルに従っているか
- [ ] 同様の処理は共通化されているか
- [ ] インポート順序は統一されているか

---

## 🔒 セキュリティ

- [ ] ユーザー入力はサニタイズされているか
- [ ] SQLインジェクション対策がされているか（Prisma使用で自動対応）
- [ ] XSS対策がされているか（Reactの自動エスケープ）
- [ ] 機密情報は環境変数で管理されているか
- [ ] ファイルアップロードは拡張子チェックがあるか

---

## 📋 レビュー時の優先度

| 優先度 | 項目 | 説明 |
|-------|------|------|
| **必須** | セキュリティ | 脆弱性のあるコードはマージ不可 |
| **必須** | 型安全性 | `any` の乱用、型エラーは修正必須 |
| **必須** | エラーハンドリング | 未処理の例外は許容しない |
| **高** | パフォーマンス | 明らかな問題は修正 |
| **高** | テスト | 新規機能はテスト必須 |
| **中** | 命名・コメント | 可読性向上のため推奨 |
| **低** | 細かなスタイル | 自動整形で対応可能な範囲 |

---

## 🚀 レビュー完了後

### Approve 条件
- すべての「必須」項目が満たされている
- 「高」優先度の項目に重大な問題がない

### Request Changes 条件
- セキュリティ上の問題がある
- 型エラーがある
- エラーハンドリングが不十分
- パフォーマンスに重大な影響がある
