# UI/UX改善ワークフロー

> **UI/UX改善の標準ワークフロー**
>
> **最重要**: 各ステップで必ずユーザー確認を取ってから進む

---

## 最重要原則：確認してから進む

### ❌ 絶対にやってはいけない

```
ユーザー: 「各カードに関してタイトルだけにしたい。サブタイトルを消したい」

AI: 「かしこまりました」→ 調べてすぐ削除（← これがダメ！）
```

### ⭕ 正しい対応

```
ユーザー: 「各カードに関してタイトルだけにしたい。サブタイトルを消したい」

AI: 「かしこまりました。確認させてください。

どのカードのサブタイトルを削除しますか？
- A. すべてのページのカード
- B. 特定のページ（起こし・NA原稿ページ）
- C. 番組選択セレクターのみ

お答えいただいてから、対象ファイルを確認し、
変更内容を提示してから実装します。」
```

## ステップバイステップ確認フロー

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     各ステップで「確認→承認→次へ」                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  Phase 1: 問題理解                                                          │
│  ├── AI: 「どの部分を変更しますか？」                                        │
│  ├── ユーザー回答                                                           │
│  └── AI: 「理解しました。○○という認識で合っていますか？」                     │
│      ↓ 【ユーザーが「はい」と言うまで進まない】                              │
│                                                                             │
│  Phase 2: 対象特定                                                          │
│  ├── AI: 「該当ファイルを調査しました。以下が変更対象です：」                 │
│  ├── AI: 「1. aaa.tsx の L12-L15」                                          │
│  │        「2. bbb.tsx の L8-L10」                                          │
│  └── AI: 「この認識で合っていますか？」                                      │
│      ↓ 【ユーザーが「はい」と言うまで進まない】                              │
│                                                                             │
│  Phase 3: 変更内容提示                                                      │
│  ├── AI: 「以下のように変更します：」                                        │
│  ├── AI: ```diff                                                            │
│  │        - <p className="subtitle">サブタイトル</p>                        │
│  │        + <!-- 削除 -->                                                   │
│  │        ```                                                               │
│  └── AI: 「この変更でよろしいでしょうか？」                                  │
│      ↓ 【ユーザーが「はい」「OK」「進めて」などと言うまで進まない】           │
│                                                                             │
│  Phase 4: 実装                                                              │
│  ├── AI: 「実装します」                                                      │
│  └── 実装完了後                                                             │
│      └── AI: 「実装完了しました。以下のファイルを変更しました：」             │
│          「- aaa.tsx: サブタイトル削除」                                     │
│          「- bbb.tsx: サブタイトル削除」                                     │
│          「ご確認ください」                                                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 各フェーズの確認ポイント

### Phase 1: 問題理解の確認

**必ず確認すること**
- [ ] どの画面・コンポーネントが対象か
- [ ] 具体的に何を変更したいか
- [ ] 「すべて」か「特定の部分」か

**確認テンプレート**
```
「ご要望を確認させてください。

【私の理解】
- 対象: ○○画面の△△コンポーネント
- 変更内容: ××を□□に変更
- 範囲: すべてのページ / 特定のページのみ

この認識で合っていますか？」
```

---

### Phase 2: 対象特定の確認

**必ず確認すること**
- [ ] 変更対象のファイル一覧
- [ ] 変更箇所の行番号
- [ ] 変更の影響範囲

**確認テンプレート**
```
「該当するファイルを調査しました。

【変更対象ファイル】
1. `app/page1.tsx` (L120-L125)
   - サブタイトル: "Premiere Pro書き起こしをAIで整形"
   
2. `app/page2.tsx` (L200-L205)
   - サブタイトル: "Zoom文字起こしをAIで整形"

【確認事項】
- これらすべてのサブタイトルを削除しますか？
- それとも特定のファイルのみですか？

お答えください。」
```

---

### Phase 3: 変更内容の確認（最重要）

**必ず確認すること**
- [ ] 変更前後の差分を提示
- [ ] 変更理由を説明
- [ ] ユーザーの承認を得る

**確認テンプレート**
```
「変更内容を提示します。

【変更予定】
```diff
// app/page1.tsx L122-L123
- <div>
-   <h1>起こし・NA原稿</h1>
-   <p className="text-sm text-gray-500">Premiere Pro書き起こしをAIで整形</p>
- </div>
+ <div>
+   <h1>起こし・NA原稿</h1>
+ </div>
```

【変更理由】
- サブタイトルを削除し、タイトルのみの表示に変更

【影響】
- 視覚的な変化: ヘッダーがよりシンプルに
- 機能的な変化: なし

この変更で実装を進めてよろしいでしょうか？
（「はい」「OK」「進めて」などの承認をお願いします）」
```

---

### Phase 4: 実装前の最終確認

**実装を開始する前に必ず言うこと**
```
「実装を開始します。以下の内容で進めます：

【実装内容】
- ファイル: ○○.tsx
- 変更: ××の削除

【完了予定時間】
- 約X分

【完了後の報告内容】
- 変更したファイル一覧
- Before/Afterの比較

よろしいでしょうか？」
```

## 確認を取らないと危険なケース

### ケース1: 「すべて」という言葉

**危険な要求**
```
ユーザー: 「すべてのカードのサブタイトルを消して」
```

**確認が必要な理由**
- 「すべて」が本当に「すべてのページ」を指すか
- 特定のカードのみかもしれない
- 意図せぬ副作用がある可能性

**正しい対応**
```
AI: 「『すべてのカード』を確認させてください。

該当しそうな箇所：
1. 起こし・NA原稿ページのヘッダー
2. 議事録・文字起こしページのヘッダー
3. 管理画面の機能カード

これらすべてのサブタイトルを削除しますか？
それとも特定の箇所のみですか？」
```

### ケース2: 曖昧な表現

**危険な要求**
```
ユーザー: 「もっとシンプルにして」
```

**確認が必要な理由**
- 「シンプル」の定義が人によって異なる
- 実装後に「違う」となりやすい

**正しい対応**
```
AI: 「『シンプルにする』を具体化させてください。

以下のどれに近いでしょうか？

A. 情報を減らす（サブタイトル削除など）
B. デザインを平らにする（影・装飾を減らす）
C. レイアウトを整理する（間隔を広げるなど）

または、参考になる画面やデザインはありますか？」
```

### ケース3: 修正・変更の要求

**危険な要求**
```
ユーザー: 「〇〇を直して」
```

**確認が必要な理由**
- 「直す」方向性が不明確
- 複数の解釈が可能

**正しい対応**
```
AI: 「〇〇を修正する方向性を確認させてください。

現在の問題：
（スクリーンショットや説明を提示）

修正案：
A. 案Aの説明
B. 案Bの説明

どちらの方向性でしょうか？」
```

## 確認のための効果的な質問

### 状況別確認質問集

#### 対象が不明確な時
```
「どの画面（ページ）を指していますか？
- A. チャット画面
- B. 起こし・NA原稿画面
- C. 議事録・文字起こし画面
- D. 管理画面
- E. その他（具体的に教えてください）」
```

#### 範囲が不明確な時
```
「『すべて』を確認させてください。

該当しそうな箇所がX件見つかりました：
1. ○○.tsx の L12
2. △△.tsx の L34
...

これらすべてを変更しますか？
それとも特定の箇所のみですか？」
```

#### 変更内容が不明確な時
```
「具体的な変更内容を確認させてください。

【私の理解】
○○の部分を△△に変更

これで合っていますか？
または、図やスクリーンショットで
イメージを教えていただけますか？」
```

#### 緊急度が不明確な時
```
「優先度を確認させてください。

- A. 今すぐ対応（他の作業を中断）
- B. 今日中
- C. 今週中
- D. 今後の改善として記録

どちらでしょうか？」
```

## 実装前チェックリスト

**以下すべてにチェックがついてから実装を開始すること**

- [ ] **対象が明確**
  - どのファイルを変更するか特定済み
  - ユーザーにファイル一覧を提示済み
  - ユーザーが「はい」と承認済み

- [ ] **変更内容が明確**
  - 変更前後の差分を提示済み
  - ユーザーが変更内容を理解済み
  - ユーザーが「はい」「OK」「進めて」と承認済み

- [ ] **影響範囲が理解されている**
  - 変更が他の部分に影響しないか確認済み
  - 必要に応じて影響範囲をユーザーに説明済み

- [ ] **実装計画を共有済み**
  - どの順序で変更するか説明済み
  - 完了予定時間を伝えている

## 失敗パターンと対策（確認不足編）

### 失敗パターン1: 「すべて」と「特定」の混同

**失敗した対応**
```
ユーザー: 「各カードに関してタイトルだけにしたい」

AI: 「かしこまりました」（→ すべてのCardコンポーネントを修正）

結果: ユーザーは「番組選択のカードのみ」を意図していた
```

**正しい対応**
```
AI: 「『各カード』を確認させてください。

以下のようなカードが見つかりました：
1. FeatureCard（機能紹介カード）
2. Card（汎用カードコンポーネント）
3. ProgramSelector（番組選択）

どのカードのサブタイトルを削除しますか？」
```

### 失敗パターン2: 確認を省略して実装

**失敗した対応**
```
ユーザー: 「サブタイトル消したい」

AI: 「かしこまりました」（→ 即座に実装）

結果: ユーザーは別のページのサブタイトルを意図していた
```

**正しい対応**
```
AI: 「サブタイトルを削除する対象を確認させてください。

サブタイトルを含む箇所：
1. transcripts/page.tsx: "Premiere Pro書き起こしをAIで整形"
2. meeting-notes/page.tsx: "Zoom文字起こしをAIで整形"
3. admin/page.tsx: 管理メニューの説明

どれを削除しますか？
またはすべて削除しますか？」
```

### 失敗パターン3: 変更内容の提示なし

**失敗した対応**
```
AI: 「かしこまりました。実装します」（→ 変更内容を提示せず実装）

結果: ユーザーは異なる変更を期待していた
```

**正しい対応**
```
AI: 「変更内容を確認させてください。

【Before】
<h1>タイトル</h1>
<p>サブタイトル</p>

【After】
<h1>タイトル</h1>
<!-- サブタイトルを削除 -->

この変更でよろしいでしょうか？
（「はい」と言っていただいてから実装します）」
```

## Phase 0: API制約調査（データ表示系のUI改善必須）

UI改善で**データの表示方法・構造**を変更する場合は、必ず先にAPI調査を行う。

### 調査チェックリスト

- [ ] **表示したい情報がAPIレスポンスに含まれているか**
  ```bash
  # 関連ドキュメントを検索
  grep -r "API|レスポンス|event" docs/ --include="*.md"
  
  # 実際のレスポンス構造を確認
  cat docs/specs/api-integration/*.md
  ```

- [ ] **親子関係（紐付け）が存在するか**
  - 例: citation と tool_call の紐付けIDはあるか？
  - 例: 複数の検索結果を個別に識別できるか？

- [ ] **同じ情報を取得する複数の方法があるか**
  - 例: `output_text.annotation.added` vs `output_item.done`
  - どちらを使うべきか、違いは何か

### 制約発見時の対応フロー

```
制約を発見
    ↓
1. 制約を明確に文書化
   「API仕様上、XXXは不可能」
    ↓
2. 代替案を3つ以上提示
   - 案A: 制約を受け入れた設計
   - 案B: 部分的に回避する設計
   - 案C: 全く別のアプローチ
    ↓
3. 各案のトレードオフを説明
    ↓
4. ユーザーに選択を委ねる
```

## Phase 1: 問題理解（対話の詳細）

### 目的
ユーザーの**言葉に隠れた本当の課題**を引き出す。

### 対話フロー

```
ユーザー: 「〇〇がきれいじゃない」
    ↓
AI: 「スクリーンショットを見せていただけますか？」
    ↓
【スクリーンショット確認】
    ↓
AI: 「ご指摘の部分、私の理解で合っていますか？
     【図示：問題箇所のマーキング】
     この4つのボックスが縦に並んでいる部分ですね？」
    ↓
ユーザー: 「はい、あと...」
    ↓
【さらに情報収集】
    ↓
AI: 「理想的にはどのような表示になりたいですか？」
    ↓
ユーザー: 「もっとまとまっている感じがいい」
    ↓
AI: 「『まとまっている』を具体化すると、
     - 1つのボックスにまとめる
     - タブで切り替える
     などが考えられます。どちらに近いイメージですか？」
```

### 確認ポイント

**このフェーズを終える前に必ず確認**
- [ ] 「この部分で合っていますか？」
- [ ] 「どの範囲を変更しますか？（すべて/特定のみ）」
- [ ] 「優先度はどの程度ですか？」

---

## Phase 2: 調査・分析（対話の詳細）

### 目的
**「技術的に可能か」**と**「どこまで改善できるか」**を明確にする。

### 対話フロー

```
【問題が具体化された】
    ↓
AI: 「技術的な調査を行います。まずAPIの仕様を確認しますね。
     10分程度お時間ください。」
    ↓
【API調査・コード確認】
    ↓
AI: 「調査結果を報告します。
     
     【期待していた構造】
     検索ごとに個別の結果を紐付けて表示
     
     【実際のAPI仕様】
     紐付け情報が存在しません
     
     【影響】
     個別の検索結果表示は不可能です
     
     代替案を提示しますか？」
    ↓
ユーザー: 「代替案を見せてください」
    ↓
【Phase 3へ進む】
```

### 確認ポイント

**このフェーズを終える前に必ず確認**
- [ ] 「調査結果の認識で合っていますか？」
- [ ] 「制約を理解していただけましたか？」
- [ ] 「代替案を提示しますか？」

---

## Phase 3: 設計・すり合わせ（対話の詳細）

### 目的
**ユーザーと認識を合わせ、進行方向に合意を得る**。

### 対話フロー

```
【調査結果が出た】
    ↓
AI: 「調査結果をもとに、3つの改善案を作成しました。
     ご確認ください。」
    ↓
【案の提示（ビジュアル付き）】
    ↓
ユーザー: 「案Aがいいけど、もう少し...」
    ↓
AI: 「ご指摘を反映しました。【案A改】
     この方向で合っていますか？」
    ↓
ユーザー: 「はい、これで進めてください」
    ↓
【合意形成完了 → Phase 4へ】
```

### 確認ポイント
