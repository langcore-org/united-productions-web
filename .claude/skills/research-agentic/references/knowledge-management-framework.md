## ナレッジ管理の種類と対象

### タイプ別管理

| タイプ | 対象 | 内容 | 頻度の目安 |
|-----------|------|---------|-----------|
| **セキュリティ監査** | 依存パッケージ、設定ファイル | CVE, セキュリティアドバイザリ | 継続的/月1回 |
| **バージョン監査** | 言語、フレームワーク、ライブラリ | 最新安定版、LTS, EOL | 四半期毎 |
| **ベストプラクティス監査** | アーキテクチャ、コーディング規約 | 業界標準、推奨パターンの変化 | 四半期毎 |
| **リスト情報監査** | 番組情報、メンバーリスト、機能一覧 | 追加・削除・変更の正確性 | 随時/変更時 |
| **手順書監査** | ドキュメント、README | 手順の正確性、スクリーンショット | 随時 |
| **スキル監査** | .claude/skills/ | 手順の有効性、スクリプトの動作 | 随時 |

### 検証対象の優先順位

```
優先度 1（必須）:
- package.json / requirements.txt / Cargo.toml など依存定義
- .env.example / 設定ファイルの機密情報扱い
- docs/specs/ 技術仕様
- ユーザーに提示するリスト型情報（番組情報等）
- SECURITY.md / セキュリティ関連ドキュメント

優先度 2（重要）:
- README.md / セットアップ手順
- docs/guides/ ガイド類
- .github/workflows/ CI/CD設定
- Dockerfile / コンテナ設定

優先度 3（参考）:
- docs/lessons/ 学びの記録
- docs/plans/ 計画書（完了後はarchiveへ）
- .claude/skills/ スキルドキュメント
- コメント・JSDoc
```

## 網羅的収集チェックリスト

### ファイルシステム走査

```bash
# ドキュメント収集
find docs -name "*.md" -type f 2>/dev/null | head -50

# 設定ファイル収集
ls -la *.{json,yaml,yml,toml,ini} 2>/dev/null
ls -la .github/**/*.{json,yaml,yml} 2>/dev/null

# 依存関係ファイル
cat package.json 2>/dev/null | jq '.dependencies, .devDependencies'
cat requirements.txt 2>/dev/null
cat Cargo.toml 2>/dev/null

# スキルファイル
ls -la .claude/skills/*/

# 環境変数設定
cat .env.example 2>/dev/null
grep -r "process.env" app/ lib/ --include="*.ts" --include="*.js" | head -20
```

### 情報抽出パターン

| 情報タイプ | 抽出パターン | 例 |
|-----------|-------------|-----|
| **バージョン** | `v?\d+\.\d+\.\d+` | v18.2.0, 2.1.0 |
| **日付** | `\d{4}[-/]\d{1,2}[-/]\d{1,2}` | 2024-01-15 |
| **URL** | `https?://[^\s]+` | https://nodejs.org |
| **パッケージ名** | `@?[^/]+/[^@]+` | @types/react, lodash |
| **環境変数** | `[A-Z_]+=` | DATABASE_URL= |
| **リスト項目** | `- ` または `\d+\.` で始まる行 | メンバーリスト、番組一覧 |

## 検証手法

### 1. 公式ソース優先原則

```
最優先:
- 公式ドキュメント (docs.{tech}.dev, {tech}.dev/docs)
- 公式ブログ / リリースノート
- GitHub Releases / CHANGELOG

次点:
- 権威ある技術メディア (MDN, Dev.to, Medium pubs)
- コアコントリビューターの記事
- 公式Discord/Forumの情報

注意:
- Stack Overflow（日付を確認）
- Qiita/Zenn（個人ブログのため検証が必要）
- 非公式翻訳（原文と照合）
```

### 2. 横断検証

1つの情報源だけに頼らず、複数ソースで確認：

```
例: React v19 のリリース確認
- React公式ブログ ✓
- GitHub Releases ✓
- npm registry ✓
- 権威ある技術メディア (The New Stack, etc.) ✓
→ 4つ以上一致で「確定」と判断
```

### 3. 日付と文脈の確認

| 確認項目 | 方法 | 重要度 |
|---------|------|-------|
| 公開日 | 記事ヘッダー、Git blame | 高 |
| 最終更新日 | Last updated, 更新履歴 | 高 |
| バージョン指定 | 「〜以上」「〜以降」の表現 | 高 |
| 前提条件 | 動作環境、依存関係の要求バージョン | 中 |

## パターン別管理手順

### A. パッケージバージョン監査

```
手順:
1. package.json の dependencies/devDependencies を抽出
2. 各パッケージの npm registry で最新版を確認
3. GitHub Releases で変更履歴を確認
4. セキュリティアドバイザリを確認 (npm audit, Snyk)
5. 破壊的変更の有無を確認
6. アップグレードの難易度を評価

出力項目:
- 現在バージョン
- 最新バージョン
- メジャー/マイナー/パッチの差分
- セキュリティリスクの有無
- 破壊的変更の要約
- 推奨アクション
```

### B. ドキュメント手順監査

```
手順:
1. README のセットアップ手順を読む
2. 手順を実際に追ってみる（頭の中でまたは実際に）
3. 不足・誤り・古い情報を特定
4. 必要なコマンドが存在するか確認
5. パス・URLが有効か確認

チェックポイント:
□ コマンドが現在のバージョンで動作するか
□ 必要な前提条件が明記されているか
□ ファイルパスが正しいか
□ 外部リンクが有効か
□ スクリーンショットが最新UIと一致するか
```

### C. スキルファイル監査

```
手順:
1. .claude/skills/{name}/SKILL.md を読む
2. 記載されている手順が現状と一致するか確認
3. scripts/ のスクリプトが動作するか検証
4. references/ の情報が最新か確認
5. 他のスキルとの重複・矛盾を確認

チェックポイント:
□ ファイルパスが正しいか
□ コマンドが現在の環境で動作するか
□ 依存パッケージがインストール済みか
□ 前提条件が明記されているか
□ エラーハンドリングが適切か
```

### D. アーキテクチャ・設計監査

```
手順:
1. docs/specs/ の設計ドキュメントを読む
2. 現在の実装と設計が一致するか確認
3. 業界のベストプラクティスと比較
4. 推奨パターンの変化を確認
5. 技術的負債の有無を評価

チェックポイント:
□ 設計と実装の整合性
□ 使用パターンが非推奨になっていないか
□ より良い代替案が出ていないか
□ パフォーマンス最適化の機会はないか
```

## リスト型情報の管理

### 対象例

- 番組情報一覧
- チームメンバーリスト
- 機能一覧・ロードマップ
- 対応言語リスト
- サポート対象ブラウザ

### リスト型情報の特徴

```
特徴:
- 頻繁に追加・削除・変更が発生
- 完全性（網羅性）が重要
- 正確性がユーザーに直接影響
- 複数箇所で同じ情報が重複するリスク
```

### 管理フロー

```
1. 現在のリストを完全に抽出
   → 重複がないか確認
   → フォーマットを統一

2. 正しい情報（一次情報源）を収集
   → 公式発表、管理画面、確定事項
   → 「正しいリスト」を明確に

3. 差分を特定
   現在: [A, B, C, D]
   正確: [A, B, D, E]
   → C を削除、E を追加

4. 削除項目の性質を判断
   □ 過去に存在したが終了したものか？
     → 「過去の実績」として別セクションに移動
   □ 誤記載か？
     → 完全に削除
   □ 判断がつかない
     → ユーザーに確認

5. 追加項目の検証
   □ 公式に発表済みか？
   □ 日付・詳細情報は正確か？
   □ 他の項目と整合性は取れているか？

6. 修正の実行
   → 差分を反映
   → 更新履歴を記録
   → 関連する他のドキュメントも更新
```

### 差分対応パターン

| パターン | 例 | 対応 |
|---------|-----|------|
| **追加** | 新番組の追加 | リストに追加、関連情報も併記 |
| **削除（終了）** | 終了した番組 | 「過去の実績」に移動、履歴として保持 |
| **削除（誤記載）** | 存在しなかった誤情報 | 完全に削除、履歴に残さない |
| **変更** | 時間変更、タイトル変更 | 現在情報を更新、変更履歴を記録 |
| **統合** | 複数リストの重複解消 | 単一情報源に統一、参照を設定 |

### 完全性チェック

```
□ リストに漏れがないか
□ 重複がないか
□ 並び順に一貫性があるか（アルファベット、日付など）
□ 必須情報が全項目に含まれているか
□ フォーマットが統一されているか
```

## 誤り検出から修正までのワークフロー

### フェーズ1: 検出

```
誤りを見つけるタイミング:
- 定期監査時
- ユーザーからの指摘
- 新情報との照合時
- コード変更時の影響調査

記録:
□ 誤りの内容
□ 誤りが見つかった場所（ファイル、行番号）
□ 正しい情報（分かっている場合）
□ 情報源
```

### フェーズ2: 影響評価

```
□ この誤りがユーザーに与える影響は？
□ 他のドキュメント・コードにも影響するか？
□ 修正の緊急度は？
□ 修正による副作用はないか？
```

### フェーズ3: 修正計画

```
1. 修正範囲の特定
   → 直接修正が必要なファイル
   → 参照・関連ファイル

2. 修正手順の立案
   → 削除・追加・変更の分類
   → 一括修正 vs 段階的修正の判断

3. バックアップ/ロールバック計画
   → 修正前の状態を把握
   → 問題があれば戻せる準備
```

### フェーズ4: 修正実行

```
□ 修正を適用
□ 関連ファイルも同時に更新
□ 更新履歴に記録（変更内容と理由）
□ コミットメッセージに「修正理由」を明記
```

### フェーズ5: 検証

```
□ 修正が正しく反映されたか確認
□ 他の部分に影響が出ていないか確認
□ フォーマット・スタイルが統一されているか確認
□ 必要に応じてユーザーに確認
```

### 修正時の判断基準

| 状況 | 判断 | 理由 |
|------|------|------|
| 誤記載を削除 | 完全削除 | 履歴に誤情報は不要 |
| 終了した項目を削除 | 「過去の実績」に移動 | 歴史的記録として価値がある |
| 古い手順を更新 | 上書き更新 | 現在有効な情報のみを保持 |
| 古い手順を残すべき | 別セクションに移動 | 旧バージョン対応等で必要な場合 |

## よくある誤りパターン

### ドキュメントの陳腐化パターン

| パターン | 例 | 検出方法 |
|---------|-----|---------|
| **バージョン陳腐化** | Node.js v16 と記載（EOL済み） | 公式サイトのリリーススケジュール |
| **手順陳腐化** | npm install で書いているが yarn 推奨に変更 | セットアップ手順の再検証 |
| **リンク切れ** | 404 になる外部URL | リンクチェッカー or 手動確認 |
| **UI不一致** | 古いバージョンのスクリーンショット | 現在のUIと比較 |
| **機能削除** | 削除されたオプションを紹介 | 変更履歴の確認 |
| **リストの不完全** | 新メンバーが追加されていない | 一次情報源との比較 |
| **日付の不一致** | 「2024年1月」と書いているが実際は2月 | カレンダー・スケジュール確認 |

### セキュリティ関連の誤り

| パターン | リスク | 対応 |
|---------|-------|------|
| **デフォルトパスワード** | セキュリティインシデント | 環境変数使用を徹底 |
| **古い暗号化方式** | 暗号化破損 | 最新推奨方式に更新 |
| **不要な権限** | 特権昇格 | 最小権限の原則を適用 |
| **ハードコードされたシークレット** | 情報漏洩 | 環境変数化 |

### コードとドキュメントの不整合

| パターン | 原因 | 対応 |
|---------|------|------|
| **関数名の不一致** | リファクタリング後の更新漏れ | コード検索で参照を確認 |
| **パラメータの不一致** | API変更後の更新漏れ | 型定義とドキュメントの比較 |
| **戻り値の不一致** | 実装変更後の更新漏れ | テストとドキュメントの比較 |
| **リスト情報の不一致** | 複数箇所の重複・更新漏れ | 単一情報源化 or 一括更新 |

## 優先度判断基準

### 重要度マトリクス

| | 影響大 | 影響小 |
|---|-------|-------|
| **確度高** | 🔴 即時対応 | 🟡 計画的対応 |
| **確度低** | 🟡 計画的対応 | 🟢 参考記録 |

### 各重要度の定義

🔴 **即時対応（Critical）**
- セキュリティ脆弱性（CVE Critical/High）
- サービス停止リスク
- コンプライアンス違反
- データ損失リスク
- **ユーザーに誤情報を提供している**

🟡 **計画的対応（Important）**
- セキュリティ警告（CVE Medium）
- パフォーマンス影響
- 非推奨化（近い将来の削除予定）
- ユーザーエクスペリエンス低下
- ドキュメントの不整合

🟢 **参考記録（Low）**
- 軽微な文言の問題
- 将来的な改善候補
- 背景情報の不足
- スタイルの問題

## ツールとスクリプト

### 推奨検証コマンド

```bash
# セキュリティスキャン
npm audit
yarn audit
snyk test

cargo audit         # Rust
pip-audit           # Python

# ライセンス確認
npm ls
license-checker     # npm

# 古いパッケージ検出
npm outdated
yarn outdated

# リスト情報の重複チェック
grep -r "番組名" docs/ --include="*.md" | sort

# 日付フォーマットの不整合チェック
grep -r "202[0-9]" docs/ --include="*.md" | grep -v "YYYY-MM-DD"
```

### カスタムスクリプト例

```bash
#!/bin/bash
# check-docs.sh - ドキュメントの基本チェック

echo "=== リンク切れチェック ==="
grep -r "http" docs/ --include="*.md" | \
  grep -oP 'https?://[^\s\)]+' | \
  while read url; do
    curl -s -o /dev/null -w "%{http_code} $url\n" "$url"
  done

echo "=== 日付の古いドキュメント ==="
find docs/ -name "*.md" -mtime +90 -exec ls -la {} \;

echo "=== TODO コメントの検索 ==="
grep -r "TODO\|FIXME\|XXX" docs/ --include="*.md" -n

echo "=== 重複リスト項目の検出 ==="
grep -r "^- " docs/ --include="*.md" | sort | uniq -d
```

## ナレッジ作成・更新のガイドライン

### 新規ナレッジ作成時

```
□ 目的が明確（検索で見つかるようタイトル・タグを設定）
□ 情報源を明示（公式ドキュメント、検証済みの情報）
□ 日付を記載（作成日、検証日）
□ 前提条件を明記
□ 他のナレッジとの参照関係を設定
□ 更新責任を明確に（誰がメンテナンスするか）
□ リスト型情報は完全性を確認
□ 重複がないか既存ナレッジを確認
```

### ナレッジ更新時

```
□ 更新履歴を記録（変更内容と理由）
□ 古い情報は削除せず「過去の記録」として適切に残すか判断
□ 関連する他のナレッジも整合性チェック
□ 最終更新日を更新
□ レビュー依頼（必要に応じて）
□ リスト変更時は差分を明確に
```

### ドキュメントライフサイクル

```
作成 → レビュー → 公開 → 定期監査 → 更新/廃止
  ↑___________________________________|
```

| ステージ | 場所 | アクション |
|---------|------|-----------|
| 作成中 | `docs/plans/` | ドラフト作成、検証 |
| 公開 | `docs/specs/` `docs/guides/` | レビュー完了、正式公開 |
| 監査対象 | `docs/` 全体 | 定期監査、更新 |
| 廃止 | `docs/archive/` | 移行完了、参照不要になったもの |

### 監査・更新完了後のチェックリスト

```
□ すべての内部ナレッジを網羅的に確認した
□ 重要な差分を重要度別に分類した
□ 各差分に対して外部情報源を明示した
□ 推奨アクションを具体化した
□ ユーザーに判断を委ねるべき項目を特定した
□ レポートを構造化して提示した
□ リスト情報の完全性を確認した
□ 重複・矛盾が解消された
□ 更新履歴を記録した
□ 関連ドキュメントの整合性を確認した
```

---

**補足**: このフレームワークは網羅的なガイドである。実際の管理では対象の性質に応じて柔軟に適用し、不要な項目はスキップして良い。重要なのは「正確性」「完全性」「一貫性」を保つこと。
