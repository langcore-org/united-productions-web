# ドメイン固有ナレッジ管理パターン

プロジェクト特有のナレッジ管理パターンとベストプラクティス。

---

## メディア/エンターテイメント業界

### テレビ番組情報管理

このプロジェクト（UPエージェント）では、テレビ番組情報をナレッジデータとして管理している。

#### 管理対象

| データタイプ | 場所 | 更新頻度 | 検証重要度 |
|-------------|------|---------|-----------|
| **番組基本情報** | `docs/united-productions-regular-programs.md` | 改編期（4月・10月） | 🔴 高 |
| **放送回データ** | `lib/knowledge/programs*.ts` | 毎週 | 🔴 高 |
| **詳細エピソード** | `lib/knowledge/programs-detailed-data.ts` | 随時 | 🔴 高 |

#### 特有の誤りパターン

| パターン | 具体例 | 検出方法 |
|---------|--------|---------|
| **放送局誤り** | 日本テレビ→フジテレビ | 公式サイト確認 |
| **曜日・時間変更** | 水曜24:15→23:15（かまいガチ） | 番組表サイト比較 |
| **シーズン終了** | レギュラー終了→不定期 | 公式発表確認 |
| **ゲスト誤り** | AI生成ゲスト名 | 複数ソース確認 |
| **タイトル誤り** | AI生成タイトル | 公式サイト確認 |
| **日付と曜日不一致** | 「3/15（火）」→実際は金曜 | カレンダー照合 |

#### 検証ワークフロー

```
1. 基本情報の確認
   → 公式サイトで番組概要を確認
   → 放送局公式番組表で時間を確認
   
2. 直近放送回の検証（5回分）
   → 公式サイトの「過去の放送」
   → 番組表.Gガイド等の番組表サイト
   → 公式SNS（X, Instagram）
   
3. 整合性チェック
   → 日付と曜日が一致するか
   → タイトルが公式表記と一致するか
   → ゲストが正しいか
   
4. 報告書作成
   → docs/verification/episode-data-verification-report.md
   → 誤りを表形式で記録
   → 修正優先度を分類
```

#### 情報源の優先順位

```
最優先（一次情報源）:
- 放送局公式サイトの番組ページ
- 番組公式サイト
- 番組公式SNS（X, Instagram, YouTube）

次点（二次情報源）:
- 番組表サイト（番組表.Gガイド、TVドガッチ等）
- Wikipedia（基本情報の補助的確認）

注意（要検証）:
- ニュース記事
- ファンサイト
- 予告情報（変更の可能性あり）
```

#### 改編期対応

| 改編期 | 時期 | 対応 |
|-------|------|------|
| **春の改編** | 4月 | 3月下旬から情報収集開始 |
| **秋の改編** | 10月 | 9月下旬から情報収集開始 |
| **特別編成** | 年末年始・お盆 | 特番情報を確認 |

#### 定期検証スケジュール

```
□ 四半期ごと（3月・6月・9月・12月）
  → 全番組の基本情報を確認
  
□ 改編期後（4月・10月）
  → 放送時間変更がないか確認
  → 新番組・終了番組の確認
  
□ 毎月
  → 放送回データの追加・修正
```

## ナレッジデータの品質基準

### 放送回データの必須項目

```typescript
interface Episode {
  id: string;           // 一意なID
  programId: string;    // 番組ID
  date: string;         // 放送日（YYYY-MM-DD）
  title: string;        // タイトル（公式表記）
  guests?: string[];    // ゲスト（公式情報）
  description?: string; // 内容概要
  verified: boolean;    // 検証済みフラグ
  verifiedAt?: string;  // 検証日
  sources: string[];    // 情報源URL
}
```

### データ品質チェックリスト

- [ ] **正確性**: 公式情報源と一致しているか
- [ ] **完全性**: 必須項目がすべて埋まっているか
- [ ] **一貫性**: 日付と曜日が一致しているか
- [ ] **鮮度**: 最新の情報か（改編期を考慮）
- [ ] **検証済み**: 検証フラグと検証日が記録されているか
- [ ] **出典**: 情報源が明示されているか

## ツールと自動化

### 推奨検証スクリプト

```bash
# 日付と曜日の整合性チェック
node scripts/verify-date-weekday.mjs

# 放送回データの重複チェック
node scripts/check-duplicate-episodes.mjs

# 過去の放送回の欠落チェック
node scripts/check-missing-episodes.mjs

# 外部リンクの有効性チェック
node scripts/check-broken-links.mjs
```

### 自動検証トリガー

```
□ データ追加時
  → サンプル検証（10-20%）
  
□ 改編期（4月・10月）
  → 全番組の基本情報検証
  
□ 週次
  → 先週の放送回データ検証
```

## トラブルシューティング

### よくある問題と対応

| 問題 | 原因 | 対応 |
|-----|------|------|
| **情報源が見つからない** | 番組ページリニューアル | アーカイブ検索、SNS確認 |
| **情報源同士で矛盾** | 更新タイミングの違い | 公式サイトを優先 |
| **過去の放送回が確認できない** | 公式サイトの過去ログ削除 | 番組表サイトで代替確認 |
| **緊急の放送変更** | 特番編成など | 直前まで再確認 |

### 検証不能時の対応

```
1. 代替情報源を探す
   → 公式SNS
   → ニュース記事
   → ファンサイト（参考程度）

2. 検証不能と明記
   → verified: false
   → notesに「未確認」と記載

3. 後日再検証
   → 次回検証時に優先的に確認
```
