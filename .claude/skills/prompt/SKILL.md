# Prompt Engineering & Tuning

> **プロンプト設計とチューニング統合スキル**

## Description

システムプロンプトの設計、改善、テストケース駆動でのチューニングを行う統合スキル。

## When to use

- システムプロンプトの新規作成
- 既存プロンプトの改善・最適化
- テストケース駆動でのチューニング
- プロンプトバージョン管理

## Source of Truth

### プロンプト管理の階層構造

```
docs/prompts/*.md          ← Single Source of Truth（人間が編集するファイル）
        ↓
.claude/skills/prompt/scripts/update-from-doc.mjs  ← DB反映用CLI
        ↓
Supabase DB                ← 本番で使用されるデータ（system_prompts / system_prompt_versions）
        ↓
アプリケーション            ← 実行時にDBからプロンプトを取得
```

### 重要ルール

1. **編集は常に `docs/prompts/*.md` を直接編集する**
   - ファイル例: `docs/prompts/MINUTES.md`, `docs/prompts/RESEARCH_CAST.md`

2. **DB反映は専用スクリプトを使用する**
   ```bash
   node .claude/skills/prompt/scripts/update-from-doc.mjs <PROMPT_KEY> "変更理由"
   ```
   - 例: `node .claude/skills/prompt/scripts/update-from-doc.mjs MINUTES "議題の詳細説明を追加"`

3. **本番で使われるのはDBのデータ**
   - アプリケーションは実行時に `system_prompts` テーブルからプロンプトを取得
   - `docs/prompts/` のファイルは直接読まれない（DB反映用のソースとして機能）

## Best Practices

### Prompt Design

- 明確な役割定義（「あなたは〜です」）
- 具体的な出力フォーマット指定
- 思考プロセスと最終出力の分離
- 例示（few-shot）の適切な使用
- 制約条件の明確化

### Testing Strategy

- シングルターンとマルチターンの両方をテスト
- 実際のユースケースに基づくテスト設計
- エッジケースの考慮
- 定量的な評価軸の設定

### Version Management

- DBでのバージョン管理
- 変更理由の記録
- 履歴の追跡可能な状態維持

## Tuning Workflow

### Step 1: セッション開始

#### 1-A. プロンプト選択

以下の番号付きメニューを表示し、番号で選択してもらいます:

```
チューニングするプロンプトを選んでください:

1. 汎用チャット (GENERAL_CHAT)
2. 議事録 (MINUTES)
3. 議事録フォーマット・会議 (MEETING_FORMAT_MEETING)
4. 議事録フォーマット・取材 (MEETING_FORMAT_INTERVIEW)
5. 出演者リサーチ (RESEARCH_CAST)
6. 場所リサーチ (RESEARCH_LOCATION)
7. 情報リサーチ (RESEARCH_INFO)
8. エビデンスリサーチ (RESEARCH_EVIDENCE)
9. 新企画立案 (PROPOSAL)
```

`\$ARGUMENTS` にキーが直接指定されている場合はメニューをスキップします。

#### 1-B. セッション初期化

選択が確定したら以下を実行します:

```bash
node .claude/skills/prompt/scripts/init-session.mjs <KEY>
```

これにより:
- DBから現在の本番プロンプト（最新バージョン）を取得
- `.claude/skills/prompt/sessions/<KEY>/draft.md` を作成（本番のコピー。既存なら継続使用）
- `.claude/skills/prompt/sessions/<KEY>/history/` を作成
- `.claude/skills/prompt/sessions/<KEY>/test-cases.md` がなければテンプレートを生成

#### 1-C. 現状分析と設定提案

`.claude/skills/prompt/sessions/<KEY>/draft.md`・`test-cases.md`・`history/` を読み込み、以下の形式でまとめて提案します。
ユーザーは各項目を確認し、変更したい箇所だけ指示します（変更なければ「OK」で確定）:

```
📋 セッション設定案

【チューニングの目的】
（draft.mdを読んで気になる点・改善余地を仮説として提示）
例: 出力フォーマットが不統一になりやすい・Web検索の指示が曖昧 など
→ この認識で合っていますか？追加・修正があれば教えてください

【テストケース】
（test-cases.mdの状態を報告）
- 既存N件あり → 「そのまま利用」を推奨 / 「見直しを推奨」（理由付き）
- 未定義 → 「新規作成が必要」
→ このままで進めますか？

【最大試行回数】
推奨: N回（理由: 現状のhistoryがN件 / 問題が〜のため）
→ 変更する場合は数字を入力

【評価軸】（draft.mdと直近historyを読んで、今回フォーカスする改善軸を提案）
- 軸は2〜4本を目安に、今回のチューニング目的に合わせて動的に設定します
- 各軸に名前（JSONのキーになる）と「5点の基準」（何ができれば満点か）を定義します
例:
  - `confirmation_flow`（確認フロー）: リサーチ前に必要情報の確認を挟めているか
  - `output_format`（出力フォーマット）: 指定のセクション・構成に沿っているか
  - `cast_quality`（人選の質）: 要件に合致した具体的な候補を提案できているか
→ 変更・追加があれば指示してください（変更なければそのまま確定）

【承認スコア閾値】
推奨: 全テスト合格（M/M）を推奨 / または N/M以上（理由付き）
→ 変更する場合は「N/M以上」の形式で入力
```

全項目の確認が取れたらStep 2へ進みます。

---

### Step 2: テストケース確認・設計

`.claude/skills/prompt/sessions/<KEY>/test-cases.md` を読み込みます。

#### 2-A. テストケースが既存の場合

内容を確認し、不十分であれば見直しを提案します。ユーザーが「そのまま使う」と言えばStep 3へ進みます。

#### 2-B. テストケースが未定義・不十分な場合

**まずナレッジベースを参照してリアルな文脈を把握します:**

```
docs/united-productions-regular-programs.md
```

このファイルには制作会社の全レギュラー番組情報が含まれています。
チューニング対象のプロンプトキーに関連する番組を特定し、
実際の番組フォーマット・MC・内容を踏まえてテストケースを設計します。

**設計の観点（3〜4件を目安）:**

| 観点 | 例 |
|------|----|
| 基本ユースケース | ジャンル指定で出演者候補を出す（シングルターン） |
| 複合要件 | 相性分析・パネル形式など追加条件あり（シングルターン） |
| トレンド重視 | SNS話題性・視聴率を意識した選定（シングルターン） |
| 対話フロー | 確認→承認→リサーチ実行の複数ターン挙動（マルチターン） |

ナレッジベースを読み込んだ後、ユーザーに確認を取らずそのままテストケースを作成・保存します。
作成後に内容を提示し、修正があれば対話して反映します。

テストケースは **シングルターン** と **マルチターン** の2種類で記述できます:

**シングルターン形式**（1回のユーザー入力に対する応答を評価）:

```markdown
## テストケース1: [名前]

### 種別
シングルターン

### 入力
[実際に想定されるユーザーの入力テキスト]

### 期待する動作
- 【必須】[ツール使用・出力内容など具体的な基準1]
- 【必須】[基準2]
- 【あると良い】[基準3]

### 重み
必須 / あると良い
```

**マルチターン形式**（複数ターンにわたる対話フローを評価）:

```markdown
## テストケース2: [名前]

### 種別
マルチターン

### 対話フロー

#### Turn 1 - ユーザー
[ユーザーの最初の入力]

#### Turn 1 - AI（期待する応答）
- 【必須】[このターンで期待する挙動（例: リサーチ前に確認を挟む）]
- 【必須】[確認項目が明確か]

#### Turn 2 - ユーザー
[ユーザーの返答（承認・追加指示など）]

#### Turn 2 - AI（期待する応答）
- 【必須】[Web検索・X検索を実行する]
- 【必須】[候補者リストを出力する]
- 【あると良い】[相性分析も含める]

### 重み
必須
```

### Step 3: 評価実行

`.claude/skills/prompt/sessions/<KEY>/draft.md` の内容をシステムプロンプトとして使い、実際のアプリ環境に近い形で各テストケースを評価します。

#### 評価の前提
- **入力** = システムプロンプト（draft.md）＋ ユーザーメッセージ
- **出力** = AIの返答テキストだけでなく、途中のツール使用（Web検索・X検索のクエリ・結果要約）も含めて評価します
- **マルチターン** の場合は、各ターンの応答（ツール使用を含む）を順に推定します

各テストケースについて以下の順で処理します:

1. **推定出力を生成**: テストケースの入力に対して応答を推定します。ツール使用の中間ステップ（「Web検索: "冷凍食品 専門家 メディア出演"」→ 結果の要約）を含めて記述してください。マルチターンの場合は、ターンごとに推定応答を生成します。
2. **セッション評価軸で採点**: Step 1-C で定義した評価軸をそのまま使い、各軸 **1〜5点** で採点して理由を記録します
3. **passed / failed を判定**: 合計点（全軸合計の66%以上 かつ 全軸3点以上）を目安に判定します

評価は **テストケースの必須基準への適合だけでなく、総合的なアウトプット品質** で判断します。
チェックリスト的な合否ではなく、「実際にこの出力を受け取ったユーザー（ADやディレクター）が満足するか」を軸にしてください。

#### 3-A. 評価軸の参考例（デフォルト）

> **注意**: 以下は参考例です。実際の評価軸は Step 1-C で毎回定義します。
> 今回チューニングしたい問題に合わせて軸の名前・基準を自由に変えてください。

以下の3軸を **それぞれ1〜5点** で採点する場合の参考基準:

---

**① 情報の正確性・信頼性（1〜5点）**

| 点数 | 基準 |
|------|------|
| 5 | 検索が実行され、固有名詞・数値・情報源がすべて具体的。誤りがなく最新情報も反映されている |
| 4 | おおむね正確。一部に出典不明・古い情報があるが実務上問題ない |
| 3 | 情報はあるが定量データ（フォロワー数など）が不足、または一部に誤りの疑いがある |
| 2 | 検索がほぼ実行されておらず、内容が曖昧・「なんとなく書いた」感がある |
| 1 | 明らかな誤り・フィクションが含まれる、またはほぼ検索なしの推測で構成されている |

定量チェックポイント:
- 検索ツールの呼び出し回数（Web/X）が指示通りか
- 人物ごとに情報源（URL・媒体名）が1件以上あるか
- X検索を指示している場合、フォロワー数や直近の投稿数値が含まれているか

---

**② 出力の構成・読みやすさ（1〜5点）**

| 点数 | 基準 |
|------|------|
| 5 | プロンプト指定のフォーマット通りで、表・箇条書きが適切。実務でそのまま使えるレベル |
| 4 | 構成はほぼ正しい。細部の整形が甘いが読みやすい |
| 3 | フォーマットに沿っているが、一部セクションが欠けていたり冗長な記述がある |
| 2 | フォーマットが崩れている、または前置きが長すぎて実用性が低い |
| 1 | 構造がバラバラで、実務では使えないレベル |

定量チェックポイント:
- 指定フォーマットのセクション（思考プロセス・情報収集・候補者・相性分析・注意事項・参考情報）が何割そろっているか
- 候補者ごとに規定の項目（プロフィール・出演実績・推奨理由・話題性）が記載されているか

---

**③ 人選・提案の質（ユーザー目線）（1〜5点）**

| 点数 | 基準 |
|------|------|
| 5 | 要件に完全に合致した人選。バリエーションがあり、推薦理由が具体的で納得感が高い |
| 4 | おおむね的確。一部に「なぜこの人？」と思う候補があるが全体として使える |
| 3 | 無難な人選だが、画一的または要件への対応が薄い部分がある |
| 2 | 要件からズレた候補が多い、または推薦理由が「専門家だから」程度で薄い |
| 1 | 要件をほぼ無視した人選、またはADが実務で使えないレベルの提案 |

定量チェックポイント:
- 候補者数が指定範囲（例: 3〜5名）に収まっているか
- 各候補の推薦理由が30字以上の具体的な記述になっているか
- ユーザーが指定した優先軸（話題性・専門性など）が選定基準に反映されているか

---

**合否の基準**

| 合計点 | 判定 |
|--------|------|
| 13〜15点 | ✅ passed（優秀） |
| 10〜12点 | ✅ passed（合格） |
| 7〜9点 | ❌ failed（改善必要） |
| 6点以下 | ❌ failed（大幅改善必要） |

いずれかの軸が **2点以下** の場合は合計点にかかわらず failed とします。

**評価が終わったら結果をJSONで渡してスクリプトに保存させます（Writeツールは使わない）:**

```bash
node .claude/skills/prompt/scripts/save-result.mjs '<JSON>'
```

JSONフォーマット:
```json
{
  "key": "RESEARCH_CAST",
  "tests": [
    {
      "name": "テストケース名",
      "passed": true,
      "input": "テストケースの入力テキスト（シングルターンはそのまま転記。マルチターンは「Turn1: [内容] / Turn2: [内容]」形式）",
      "output": "推定した応答の要約。ツール使用の中間ステップ（例: 「Web検索: '冷凍食品 専門家 テレビ出演'→〇名を発見」）と主要な出力内容を含める。マルチターンの場合はターンごとの応答を区切って記述。",
      "scores": {
        "Step1-Cで定義した軸名": { "score": 4, "reason": "なぜこの点数なのかの具体的な理由" },
        "Step1-Cで定義した軸名2": { "score": 5, "reason": "なぜこの点数なのかの具体的な理由" }
      },
      "note": "総合コメント・次の改善点など"
    },
    {
      "name": "テストケース名",
      "passed": false,
      "input": "テストケースの入力テキスト",
      "output": "推定出力の要約",
      "scores": {
        "Step1-Cで定義した軸名": { "score": 2, "reason": "失敗した具体的な理由" },
        "Step1-Cで定義した軸名2": { "score": 3, "reason": "評価理由" }
      },
      "note": "失敗した原因と改善すべき点"
    }
  ]
}
```

> **scores のキー**: Step 1-C で定義した評価軸の名前（英語・日本語どちらでも可）をそのまま使います。固定名（accuracy/structure/quality）に縛られません。

スクリプトが自動で:
- スコアを計算
- `.claude/skills/prompt/sessions/<KEY>/history/attempt-N_score-X-Y.md` に入力・推定出力・評価理由付きで保存

---

### Step 4: 問題分析・改善案提案

失敗したテストケースを分析し、draft.mdのどの部分が原因かを特定して改善案を提示します。

ユーザーが承認 or 修正指示を出したら `.claude/skills/prompt/sessions/<KEY>/draft.md` をEditツールで更新します。

**Step 3 〜 Step 4 を繰り返します。** ただし以下の条件でユーザーに確認します:

- 承認スコア閾値に到達 → 「スコアがN/Mに達しました。本番に反映しますか？」
- 最大試行回数に到達 → 「上限（N回）に達しました。現状で本番反映しますか？それとも上限を延長しますか？」
- いずれもユーザーが「続ける」と言えばループを継続

---

### Step 5: 本番反映

ユーザーが「承認」「本番に反映」「approve」などの意思を示したら実行します:

```bash
node .claude/skills/prompt/scripts/approve.mjs <KEY> "<変更理由のメモ>"
```

これにより:
- `draft.md` の内容がDBに新バージョンとして保存される
- `.claude/skills/prompt/sessions/<KEY>/history/` 内の最後の試行ファイルに `_APPROVED` が付く
