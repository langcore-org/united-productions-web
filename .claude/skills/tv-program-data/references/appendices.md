## 付録A: 今回の検証作業から得られた教訓

### 2026年2月27日 放送回データ検証

#### 作業概要
- **対象**: 13番組×5回分=65件の放送回データ
- **検証方法**: xAI API (Grok) + Web検索
- **結果**: 大部分のデータに誤りが発見

#### 発見された誤りパターン

| 誤りタイプ | 件数 | 具体例 |
|-----------|------|--------|
| 放送局誤り | 6番組 | 日本テレビ→フジテレビ、Eテレ→総合など |
| 曜日・時刻誤り | 4番組 | 水曜→日曜、24:15→23:15など |
| タイトル創作 | ほぼ全件 | AIが生成した架空の詳細タイトル |
| 存在しない放送回 | 複数 | 実際には放送されていない日付 |

#### 根本原因分析

1. **検証前コミット**
   - データ作成→コミット→検証の流れ
   - 誤情報がAIに提供されるリスク

2. **一次情報源未確認**
   - 公式サイトでの確認が不十分
   - AI生成データをそのまま使用

3. **曜日・日付整合性チェック欠如**
   - 日付と曜日が一致していないケース多数
   - カレンダー照合が行われていない

#### 改善策（実施済み）

1. **即座の無効化**
   ```typescript
   // 誤りが見つかったため一時的に無効化（2026-02-27）
   recentEpisodes: [],
   ```

2. **検証プロセス確立**
   - サンプル検証（10-20%）を必須化
   - 公式サイトからの収集テンプレート作成
   - 検証チェックリスト作成

3. **スキルドキュメント改善**
   - 「データは嘘をつく」原則を追加
   - TV番組データ検証ガイドを追加
   - 緊急時対応ワークフローを追加

#### 今後の運用ルール

```
データ作成 → サンプル検証（10-20%）→ 問題発見 → 修正
→ 全件検証または信頼できるソース確認 → コミット
```

## 付録C: 大規模データ修正の実践パターン

### パターン1: 段階的修正アプローチ

大量の誤りデータ（13番組×5回分=65件など）を修正する場合：

```
❌ 非効率なアプローチ:
全件を一度に修正 → 時間がかかりすぎる → ミスが増える → 検証が不十分

✅ 推奨アプローチ:
1. 優先度分類
   🔴 放送局誤り（ユーザーに直接影響）
   🟡 タイトル誤り（信頼性低下）
   🟢 軽微な表記違い

2. カテゴリ別に修正
   - 放送局のみ先に修正（影響が大きいため）
   - 放送回データは番組ごとに順次修正
   - 修正済みのものから順次コミット

3. 中間コミット
   - 3-4番組修正するごとにコミット
   - 作業記録を残す
   - 進捗を明確にする
```

### パターン2: Subagent並行処理

複数の番組を同時に修正する場合：

```
親エージェント:
  ├─ Subagent1: マツコの知らない世界を調査・修正
  ├─ Subagent2: しくじり先生を調査・修正
  ├─ Subagent3: かまいガチを調査・修正
  └─ ...

注意点:
- 各Subagentは独立したファイルを担当
- 同じファイルを同時に編集しない
- 結果を統合してコミット
```

### パターン3: 検証結果の信頼性評価

xAI APIなどの自動検証結果と実際の情報が異なる場合：

```
状況: xAI検証で「誤り」と判定されたが、実際は正しかった

対応フロー:
1. 一次情報源（公式サイト）で確認
2. xAIの検証結果は「参考情報」として扱う
3. 公式サイトと矛盾する場合、公式サイトを優先
4. 判断に迷ったらユーザーに確認

教訓:
- 自動検証は100%正確ではない
- 常に一次情報源での最終確認が必要
- 「誤り」が見つかっても、過剰修正に注意
```

## 付録E: 今回の作業から得られた追加教訓（2026-02-27〜）

### 教訓1: 検証結果と実際の乖離

**状況**: xAI API検証で「誤り」と判定されたが、公式サイト確認で正しかった

**例**: しくじり先生
- xAI検証: 「2026-01-02 藤田ニコル新婚SPは不存在」
- 実際: 「お笑い研究部 エバース前半戦」が正しく放送されていた

**対応**:
- xAI検証は参考情報として扱う
- 必ず一次情報源（公式サイト）で最終確認
- 自動検証の「誤り」判定に踊らされない

### 教訓2: 段階的コミットの重要性

**良かった点**:
- 3-4番組修正ごとにコミットしたため、進捗が明確
- 問題発生時の巻き戻しが容易
- 作業記録が残り、振り返りが可能

**改善点**:
- もう少し細かくコミットしても良かった（1-2番組ごと）
- コミットメッセージに「修正済み/未済み」を明記

### 教訓3: Subagentの活用法

**効果的だった使い方**:
- 放送局修正（調査系）→ Subagentに委任
- 複数番組の並行調査が可能

**注意点**:
- 同じファイルを複数Subagentで編集しない
- 結果の統合は親エージェントで管理
- 各Subagentの結果を個別に検証

### 教訓4: 「すでに修正済み」の発見

**状況**: 修正しようとしたら、すでに正しい値になっていた

**例**: 放送局情報
- 千鳥の鬼レンチャン: すでに「フジテレビ系列」
- 熱狂マニアさん！: すでに「TBS系列」

**学び**:
- 修正前に必ず現状確認
- 無駄な作業を避ける
- 検証レポートは「時点」の情報であり、最新状況とは異なる可能性
