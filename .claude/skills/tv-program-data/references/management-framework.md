# ナレッジ管理フレームワーク

Teddyがチャットで参照するナレッジデータ（`lib/knowledge/`）の管理・更新・検証のための詳細ガイド。

## 管理対象

```
lib/knowledge/
├── programs.ts                    # 番組基本情報
├── programs-detailed-data.ts      # 番組詳細・放送回データ
├── programs-detailed-data-2.ts    # 追加放送回データ
├── talents.ts                     # タレント情報
└── [その他のナレッジデータ]
```

## ナレッジタイプ別管理

### Type A: 番組基本情報

**ファイル**: `programs.ts`

```typescript
interface Program {
  id: string;
  name: string;
  broadcaster: string;    // 放送局
  schedule: string;       // 放送時間
  cast: string[];         // 出演者
  description: string;    // 番組概要
}
```

**更新頻度**: 改編期（4月・10月）+ 随時
**検証重要度**: 🔴 高

**検証ポイント**:
- [ ] 放送局が正しい
- [ ] 曜日・時間が正しい
- [ ] MC/出演者が最新
- [ ] 番組状態（放送中/終了）

### Type B: 放送回データ

**ファイル**: `programs-detailed-data.ts`, `programs-detailed-data-2.ts`

```typescript
interface Episode {
  broadcastDate: string;  // 放送日
  title: string;          // タイトル
  guests?: string[];      // ゲスト
  content?: string;       // 内容
}
```

**更新頻度**: 毎週（放送後）
**検証重要度**: 🔴 高

**特有の誤りリスク**:
- 日付と曜日の不一致
- AI生成タイトル（架空の内容）
- 存在しない放送回
- ゲスト名の誤り

### Type C: タレント情報

**ファイル**: `talents.ts`

**更新頻度**: 随時（所属変更、新番組出演時）
**検証重要度**: 🟡 中〜高

## リスト型情報の管理

### 番組リスト管理のポイント

```
特徴:
- 頻繁に追加・削除・変更が発生（改編期）
- 完全性（網羅性）が重要
- 正確性がユーザーに直接影響
- 複数ファイルで重複のリスク
```

### 改編期対応（4月・10月）

```
1. 情報収集（3月下旬・9月下旬開始）
   → 各放送局の改編発表を確認
   → 新番組・終了番組の一覧を作成

2. 差分検出
   現在リスト: [A, B, C, D]
   新シーズン: [A, B, D, E]
   → C を「過去の番組」に移動
   → E を新規追加

3. 終了番組の扱い
   → 「過去の番組」セクションに移動
   → 完全削除はしない（履歴として保持）

4. 時間変更の確認
   → 継続番組の放送時間が変更されていないか
   → 曜日変更も確認
```

### 差分対応パターン

| パターン | 具体例 | 対応 |
|---------|--------|------|
| **新規追加** | 新番組開始 | `programs.ts` に追加 |
| **時間変更** | 水曜24:15→23:15 | 該当項目を更新 |
| **曜日変更** | 火曜→月曜 | 該当項目を更新 |
| **終了** | レギュラー終了 | 「過去の番組」に移動 |
| **一時休止** | シーズン終了 | `status: 'season_end'` 等を設定 |

## 誤り検出から修正まで

### フェーズ1: 検出

```
誤りを見つけるタイミング:
- 定期検証時（四半期ごと）
- 改編期（4月・10月）
- ユーザーからの指摘
- 新規データ追加時のサンプル検証

記録:
□ 誤りの内容
□ 誤りが見つかったファイル・場所
□ 正しい情報（分かっている場合）
□ 情報源
```

### フェーズ2: 影響評価

```
□ この誤りがユーザーに与える影響は？
  → 「放送局が違う」→🔴 重大（ユーザーが見逃す）
  → 「タイトルが違う」→🟡 中（情報の信頼性低下）
  → 「ゲスト名の表記違い」→🟢 低（軽微）

□ 修正の緊急度は？
  → 🔴 即時（誤情報を拡散させない）
  → 🟡 計画的（次回更新時）
```

### フェーズ3: 修正実行

```
1. 正しい情報の収集
   → 一次情報源（公式サイト）で確認
   → 複数ソースで横断検証

2. 修正の適用
   → 該当ファイルを更新
   → 関連ファイルも同時に確認
     （programs.ts と detailed-data.ts の整合性）

3. 更新履歴の記録
   → 修正内容をコメントまたは別ファイルに記録
   → 検証日を記録
```

### フェーズ4: 検証

```
□ 修正が正しく反映されたか確認
□ 他の部分に影響が出ていないか確認
□ フォーマットが統一されているか確認
□ 必要に応じてユーザーに確認
```

## 更新ワークフロー

### 新規データ追加時

```
❌ 誤った流れ:
データ作成 → コミット → 検証 → 誤り発見 → 修正コミット
→ Teddyが誤情報を学習・提供するリスク

✅ 正しい流れ:
データ作成 → サンプル検証（10-20%）→ 問題発見 → 修正
→ 全件検証または信頼できるソース確認 → コミット
→ Teddyが正確な情報を学習
```

### 定期更新サイクル

```
□ 毎週（放送後）
  → 先週の放送回データを追加
  → サンプル検証（最新2-3回）

□ 毎月
  → 累積データの品質確認
  → 欠落データの確認

□ 四半期ごと（3月・6月・9月・12月）
  → 全番組基本情報の確認
  → 時間変更がないか確認

□ 改編期（4月・10月）
  → 新番組・終了番組の対応
  → 継続番組の時間変更確認
  → 全データの見直し
```

## 品質基準とチェックリスト

### データ投入前の必須チェック

```
□ Primary Source（一次情報源）は確認したか？
  - 放送局公式サイト
  - 番組公式サイト
  - 番組公式SNS

□ サンプル検証（10-20%）を実施したか？

□ 時系列データは特に注意したか？
  - 日付と曜日の整合性
  - 存在しない日付がないか

□ 関連ファイルの整合性は確認したか？
  - programs.ts
  - detailed-data.ts
  - detailed-data-2.ts

□ 削除・変更項目の扱いは適切か？
  - 終了→「過去の番組」に移動
  - 誤記載→完全削除
```

### 検証記録の保存

```markdown
## 検証記録
- **検証日**: YYYY-MM-DD
- **検証対象**: [ファイル名]
- **検証方法**: [公式サイト、番組表等]
- **検証結果**: 
  - ✅ 正確: X件
  - ⚠️ 一部誤り: Y件
  - ❌ 誤り: Z件
- **修正内容**: [修正した項目]
- **未検証項目**: [次回以降に確認]
```

### 優先度判断基準

| 重要度 | 状況 | 対応期限 |
|-------|------|---------|
| 🔴 **即時** | 放送局誤り、時間誤り、存在しないデータ | 当日中 |
| 🟡 **計画的** | タイトル誤り、ゲスト誤り、軽微な表記違い | 1週間以内 |
| 🟢 **参考** | 説明文の改善、追加情報 | 次回更新時 |

## 実例：番組情報更新

```
状況: ユーザー「かまいガチの放送時間を更新して」

1. 現在データ確認
   → programs.ts: "毎週水曜 24:15〜"

2. 正しい情報収集
   → テレビ朝日公式サイト確認
   → 「毎週水曜 よる11:15～」と記載
   → 2022年4月改編で24:15→23:15に変更

3. 差分特定
   → 24:15→23:15（1時間繰り上がり）

4. 修正実行
   → programs.ts を更新
   → detailed-data.ts も確認（影響なし）

5. 検証記録
   → docs/lessons/ に検証記録を作成
   → 変更履歴に記録

6. 完了報告
   → 修正内容をユーザーに報告
   → 変更理由（2022年4月改編）も説明
```
