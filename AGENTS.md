## 🔄 スキルの自律的改善（重要）

### 基本原則

**スキルを使用する際、常に改善の機会を探し、発見したら即座に更新する。**

```
使用 → 改善点発見 → 即座に更新 → 再使用 → さらなる改善
```

### 改善を行うタイミング

以下の状況で**必ず**スキルを改善する：

| 状況 | 改善内容 | 更新先 |
|------|---------|--------|
| 「この手順がない」 | 新しい手順を追加 | `SKILL.md` |
| 「このスクリプトがほしい」 | 新スクリプト作成 | `scripts/` |
| 「説明がわかりにくい」 | ドキュメントを書き換え | `SKILL.md` |
| 「エラーが出た」 | トラブルシューティング追加 | `SKILL.md` |
| 「もっと効率的な方法がある」 | ベストプラクティス更新 | `SKILL.md` |

### 改善プロセス

```bash
# 1. 改善点を発見
スキル使用中に「あ、こうすればもっといい」と思った

# 2. 即座に更新
SKILL.md や scripts を編集

# 3. 動作確認
更新したスクリプトが動作するか確認

# 4. コミット（重要）
git add .claude/skills/[skill-name]/
git commit -m "refactor(skill): improve [skill-name] - [改善内容]"

# 5. 継続的に使用・改善
```

### 具体例

**例1: スクリプトが不足している**
```
使用中: 「あ、この確認を自動化するスクリプトがあればいいな」
↓
即座に作成: scripts/check-xxx.mjs
↓
SKILL.md に追加: 「このスクリプトで確認できます」
↓
コミット
```

**例2: ドキュメントが不完全**
```
使用中: 「このエラーの対処法が書いてない」
↓
即座に追加: SKILL.md に「トラブルシューティング」セクション追加
↓
コミット
```

**例3: 手順が古い**
```
使用中: 「この手順、もっと簡単な方法がある」
↓
即座に更新: 手順を書き換え
↓
コミット
```

### 重要: 完璧を待たない

**「大きな改善をまとめてやろう」ではなく「小さな改善を即座にやる」**

- ✅ 1つのスクリプト追加だけでもコミット
- ✅ 1行のドキュメント修正だけでもコミット
- ✅ タイポ修正だけでもコミット

**小さな改善の積み重ねが、スキルの質を上げる。**

> **発見日**: $(date +"%Y-%m-%d")

EOF

# コードに TODO コメントを残す
# TODO: [優先度] 説明
# docs/backlog/todo-xxx.md を参照
```

詳細: [docs/backlog/README.md](./docs/backlog/README.md)

```

├── README.md          # 入り口（必ず最初に読む）

## 🎯 タスク着手前の必須確認

コードを変更する前に、必ず以下を確認してから作業を始めること。

### 1. タスクスコープの定義
- 変更するファイルを着手前にリストアップし、ユーザーに提示する
- **関連する根本問題**がスコープ外ファイルにある場合は、併せて対応を検討
- 技術設計に関わる問題（DBスキーマ、API設計など）を発見した場合は、理想的な設計を提案する

```
# 着手前に宣言する例
## 変更予定ファイル
- app/(authenticated)/transcripts/page.tsx（重複定義の削除）

## 併せて検討する可能性のあるファイル
- lib/db/schema.ts（重複の原因となっているスキーマ設計）
```

### 2. plans/ の確認（実装タスクの場合）

実装を開始する前に、必ず `plans/` に計画書が存在することを確認：

```bash
# 現在の計画を確認
ls -la docs/plans/

# 関連する計画書を読む
cat docs/plans/{関連する計画}.md
```

**計画書がない場合は作成するか、ユーザーに確認する。**

### 3. 環境・依存関係の事前調査
エラーが出てから調べるのではなく、**変更前に**確認する。

```bash
# パッケージの実際のバージョンを確認する
cat package.json | grep -A1 "dependencies"
cat node_modules/<package>/package.json | grep '"version"'

# 型定義の実際のエクスポートを確認する
grep -n "export" node_modules/<package>/index.d.ts
```

## 🔒 セキュリティ・設定ファイルの扱い

### どんな時に参照するか

| 状況 | 対応 |
|------|------|
| **「型エラーが出るから認証設定を削除したい」** | 削除せず、型アサーションで対応（認証が壊れるため） |
| **「CORS/CSRF設定を変更したい」** | 変更前に意図を確認（セキュリティに影響する） |

### 削除してはならない設定

```typescript
// ❌ 型エラーが出るからといって削除してはならない
trustHost: process.env.VERCEL_ENV === "preview" || ...
// → Vercel プレビュー環境の認証に必要

// ❌ 理解せずに削除してはならない
NEXTAUTH_SECRET, AUTH_SECRET, CSRF設定, CORSヘッダー
```

### 型エラーでの対処

```typescript
// ✅ 正しい対処: 型アサーションを使用
const options = {
  ...authOptions,
  trustHost: true,
} as AuthOptions & { trustHost?: boolean }
```

## 🔁 デバッグ・修正の設計見直し

### どんな時に報告するか

| 状況 | 対応 |
|------|------|
| **「同じファイルを3回以上修正している」** | **設計見直しを提案** - 根本的な問題がある可能性 |
| **「lintエラーを無視コメントで回避した」** | `docs/backlog/` に記録し、後で根本解決 |
| **「--no-verify でコミットした」** | 必ず backlog/ にTODOを作成 |

### 同じファイルを3回以上修正した場合

これは**設計に根本的な問題があるサイン**です：
1. **立ち止まって設計を見直す**（15-30分）
2. **リファクタリング案をユーザーに提案する**
3. 無理に機能追加を続けるより、一度整理を優先することも検討

### lintエラーの対応方針

#### 根本的な解決を優先する
lintエラー（biome/eslint等）が発生した場合：

1. **最初から根本的な解決を試みる**
   - 無視コメント（ignore）での回避は最後の手段
   - useCallback/useMemoの適切な使用
   - 依存配列の正しい設定
   - **必要に応じて設計を見直す**

2. **一時的な回避が必要な場合**
   - `--no-verify`でのコミットは一時的な対応
   - **必ず`docs/backlog/`に記録する**
   ```bash
   cat > "docs/backlog/todo-$(date +%s)-lint-error-description.md" << 'EOF'
   > **優先度**: ⏸️ 保留
   > **発見日": $(date +"%Y-%m-%d")
   > **関連ファイル": app/path/to/file.ts
   
   # TODO: lintエラーの根本的解決
   
   ## 内容
   
   一時的に無視コメント/--no-verifyで回避したlintエラーの詳細
   
   ## 本来の対応
   
   - 根本的な解決方法を記載
   
   ## 先送り理由
   
   - 先送りした理由を記載
   EOF
   ```

3. **先送り禁止のケース**
   - セキュリティ関連の設定
   - 既存コードのlintエラー（今回の修正で発生したもの）
   - 型安全性を損なう変更

```
# 報告フォーマット
## 現状報告
- 修正済み: XXX（完了）
- 進行中: auth-options.ts の next-auth 型問題（3回目の修正）

## 問題の詳細
next-auth のインストールバージョン (v4.24.13) と
型定義の間に不整合があります。

## 推奨対応
型アサーションで回避するか、モジュール拡張で根本解決するかを選択してください。
```

### 試行錯誤を減らす調査ファースト原則
修正を試みる前に、必ずドキュメント・型定義・実際のコードを読む。

```bash
# 例: next-auth の型を調べてから修正する
grep -rn "export.*Options\|export.*Config" node_modules/next-auth/*.d.ts
grep -rn "export.*Session" node_modules/next-auth/core/types.d.ts
```

## 🏗️ 技術設計における理想的設計の優先

### 原則

DBスキーマ、アーキテクチャ、API設計などの**技術設計を変更する場合**:

- **「小さく済ます」修正を選ばない**
- 毎回「**理想的な設計は何か**」を考える
- 技術的負債を残す判断は、ユーザーと明示的に合意した場合のみ

### 具体例

| ❌ 避けるべき対応 | ⭕ 推奨される対応 |
|-----------------|-----------------|
| 「とりあえずカラムを追加」 | 正規化を見直し、適切なリレーションを設計 |
| 「既存APIにパラメータを追加」 | RESTfulな設計で新規エンドポイントを検討 |
| 「既存テーブルを改修」 | マイグレーション戦略を含めた設計を提案 |
| 「ワークアラウンドで回避」 | 根本原因を解決し、再発を防ぐ |

### 判断フロー

```
問題を発見したとき:

1. これは技術設計に関わる問題か？
   → Yes: 理想的な設計を考え、小さな修正は避ける

2. 修正工数の見積もり:
   - 4時間以内 → その場で理想的な設計を実装
   - 4時間以上 → ユーザーに判断を仰ぐ（理想的設計案を提示）

3. ワークアラウンドを選ぶ場合:
   - 技術設計の問題では「後で対応」は原則認めない
   - どうしても避けられない場合は、即座にリファクタリング計画を立てる
```

## ⚠️ 根本解決を必須とするケース

以下の場合は、軽微な修正ではなく**根本的な解決**を行う:

| カテゴリ | 具体例 |
|---------|--------|
| **DB設計** | スキーマ変更、マイグレーション、正規化の見直し |
| **API設計** | エンドポイント設計、レスポンス形式の統一 |
| **セキュリティ** | XSS、SQLインジェクション、認証・認可の問題 |
| **パフォーマンス** | N+1問題、インデックス設計、キャッシュ戦略 |
| **型安全性** | `as any` の使用、型定義の不整合 |
| **コード品質** | 同じコンポーネントで繰り返しバグが発生、lint無視コメントが必要になる変更 |

### 技術的負債を意図的に選ぶ場合

上記に該当する問題を「後回し」にする場合:
1. **リスクを明確に文書化**
2. **解消期限を設定**（原則3ヶ月以内）
3. **ユーザーと明示的に合意**を取る

## 📝 型安全性の向上パターン

### `as any` の除去方法

型安全性を損なう `as any` は以下の方法で段階的に除去する。

#### 方法1: モジュール拡張（推奨）
外部ライブラリの型が不足している場合、`types/` ディレクトリで拡張する。

```typescript
// types/next-auth.d.ts
import type { DefaultSession } from "next-auth";

declare module "next-auth" {
  interface Session extends DefaultSession {
    user: {
      id: string;
      role?: string;
    } & DefaultSession["user"];
    accessToken?: string;
  }
}
```

#### 方法2: 明示的なインターフェース定義
動的インポートの戻り値に型を付ける。

```typescript
// ❌ 禁止
const pdfParse: any = await import("pdf-parse");

// ✅ 正しい
interface PDFParseModule {
  default: (buffer: Buffer) => Promise<{ text: string; numpages: number }>;
}
const pdfParse = await import("pdf-parse") as unknown as PDFParseModule;
```

#### 方法3: 型ガード関数
型の絞り込みに使用する。

```typescript
function isAuthenticatedSession(session: Session | null): session is Session & { user: { id: string } } {
  return !!session?.user?.id;
}
```

### 実績
- **2026-02-19**: コードベースから `as any` を完全除去（17箇所 → 0箇所）
- **手法**: モジュール拡張 + 明示的インターフェース
- **結果**: ビルド成功、型安全性向上

| **「ビルドが通るか確認したい」** | `npm run build` |
| **「as any を使っていないか確認したい」** | `git diff --staged \| grep "as any"` |

すべて pass してからコミットすること。

```bash
# 実行例
npx tsc --noEmit 2>&1 | grep -v node_modules  # 型チェック
npm run lint                                   # Lint
npm run build                                  # ビルド
git diff --staged | grep "as any"             # as any チェック（出力があれば修正）
```

- 変更したファイルと理由
- 発見した既存問題とその対応状況

## 🚨 報告条件

### どんな時に報告するか

| 状況 | 報告内容 |
|------|---------|
| **「同じファイルを3回以上修正している」** | **設計見直しの提案** |

| **「認証/CORS/シークレット設定を変更する必要がある」** | 意図確認を報告 |

| **「技術的負債を残す判断が必要な場合」** | リスクと解消計画を報告 |

### 深い問題を発見した場合

設計の問題や根本的な解決が必要な場合：
1. **立ち止まって考える時間を設ける**（15-30分）
2. 軽微な修正と根本解決の**両方の選択肢を提示**
3. **推奨する方針を明確に述べる**
4. ユーザーの判断を仰ぐ

## 📁 ドキュメント運用ルール

### どんな時にどこに配置するか

| 状況 | 配置先 | 命名規則 |
|------|--------|---------|
| **「アイデアをメモしたい」** | `docs/backlog/` | `idea-{機能名}.md` |
| **「調査・検討を記録したい」** | `docs/backlog/` | `research-{調査名}.md` |
| **「実装計画を立てたい」** | `docs/plans/` | `{計画名}.md` |
| **「実装が完了した」** | `docs/archive/` | `YYYY-MM-DD-{元のファイル名}.md` |
| **「教訓・知見を記録したい」** | `docs/lessons/` | `YYYY-MM-DD-{タイトル}.md` |

### ファイル命名規則の詳細

| ディレクトリ | 命名規則 | 例 |
|-------------|---------|-----|
| `docs/plans/` | `計画名.md`（プラン作成日はメタデータ内） | `conversation-memory-design.md` |
| `docs/lessons/` | `YYYY-MM-DD-タイトル.md`（**終了日**を先頭に） | `2026-02-26-chat-streaming-loading-issue.md` |
| `docs/archive/` | `YYYY-MM-DD-元のファイル名.md`（**アーカイブ日**を先頭に） | `2026-02-22-refactoring-completed.md` |
| `docs/backlog/` | `idea-機能名.md` / `research-調査名.md` / `todo-タスク名.md` | `idea-vtuber-research.md` |

**重要**: 
- `lessons/` は**終了・完了した日付**をファイル名に入れる
- `archive/` は**アーカイブした日付**をファイル名に入れる
- `plans/` は**作成日ではなく計画名**をファイル名に入れ、日付はメタデータ内に記載

### ドキュメントライフサイクル

```
【実装前】
アイデア発生 → backlog/ に配置 → 検討・調査 → 実装決定 → plans/ に計画書作成

【実装中】
plans/ の計画書を随時更新（実装状況を記録）
    ↓ 保留タスク・イシュー発見 → backlog/ に記録

【実装完了後】
        ↓
「教訓・知見が得られたか？」
    ↓
    Yes → lessons/ に変換・移動（終了日をファイル名に）
    ↓
    No  → archive/ に移動（アーカイブ日をファイル名に）
```

### 判断基準: lessons/ への移動

以下のいずれかに該当する場合は `lessons/` に移動：

- 「3ヶ月後の自分が同じ状況で、これを知っていれば異なる判断をするか」→ Yes
- 技術選定の失敗・成功
- アーキテクチャの変更
- パフォーマンス改善の知見
- フレームワーク・ライブラリの導入判断
- 同じ問題で3回以上試行錯誤した
- 見積もりが2倍以上外れた

### lessons/ への変換手順

```bash
# 1. 計画書をベースに lessons ファイルを作成（終了日を先頭に）
cp docs/plans/your-plan.md "docs/lessons/$(date +%Y-%m-%d)-title.md"

# 2. 内容を書き換え（タスク一覧→結果サマリー、技術選定理由→背景・判断基準）
# 3. 必須セクション: 概要、背景、結果、教訓、推奨事項
# 4. lessons/README.md に一覧追加
# 5. 元の計画書は削除（または archive/ に移動して参照先を記載）
```

### 各ディレクトリの役割

| ディレクトリ | 役割 | 更新頻度 | 信頼度 |
|-------------|------|---------|-------|
| `specs/` | 技術仕様（唯一の信頼情報源） | 随時 | ⭐⭐⭐ |
